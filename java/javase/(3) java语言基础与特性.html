03 java语言基础与特性

jiwang4/16/26About 23 minjavase

一、迭代器

1.1 什么是迭代器？

迭代器是一种行为型设计模式。它的核心作用是：提供一种方法顺序访问一个容器对象中的各个元素，而又不需要暴露该对象的内部表示。

在 Java 集合框架中，无论底层是数组（ArrayList）、链表（LinkedList）还是散列表（HashMap），我们都可以通过统一的 Iterator 接口进行遍历。

1.2 为什么要用迭代器

如果你直接使用 for(int i=0; i<list.size(); i++)：

  • 依赖具体实现：你必须知道容器是否有序、是否有索引。
  • 扩展性差：如果有一天从 ArrayList 换成了 Set，代码就得大改。

迭代器将“遍历逻辑”从“容器实现”中抽离了出来。

1.3 核心机制：游标与状态

迭代器通常维护一个指向当前元素的游标（Cursor）。主要操作只有三个：

  • hasNext()：判断后面是否还有元素。
  • next()：返回当前指向的元素，并将游标后移。
  • remove()：安全地删除刚刚返回的元素。

1.4 Fail-Fast

在 Java 中，如果你在使用迭代器遍历集合的过程中，通过集合对象本身（如 list.remove()）修改了集合长度，迭代器会立即抛出 ConcurrentModificationException。

底层逻辑： 集合内部维护一个 modCount（修改次数）。迭代器创建时会记录当时的 expectedModCount。每次调用 next() 都会检查两者是否相等，如果不等，说明集合被“偷摸”改动了。

最佳实践： 若需在遍历时删除元素，请务必使用 iterator.remove()，它会在删除后同步更新 expectedModCount。

二、泛型

2.1 什么是泛型

简单来说，泛型就是把“类型”也当成参数传递。 在没有泛型之前，我们写一个方法或类，里面操作的数据类型是写死的。有了泛型，你可以像传递变量 x = 10 一样，传递类型 <T = String>。

2.2 为什么要用泛型

在泛型出现之前，集合（比如 List）默认装的都是 Object。

  • 存数据时：什么都可以往里塞（猫、狗、字符串），编译器不管。
  • 取数据时：你必须进行强制类型转换。一旦你误把狗当成猫强转，代码在运行期就会直接崩溃，抛出 ClassCastException。

泛型的核心价值在于“类型安全”和“消除强转”：它把运行期的报错，提前到了编译期。你往 <String> 的集合里塞 Integer，IDE 直接标红，代码根本跑不起来。

2.3 类型擦除 (Type Erasure)

像 Java 这种语言，泛型其实是伪泛型。泛型信息只存在于编译阶段。一旦代码被编译成字节码进入 JVM，所有的泛型 <T> 都会被“擦除”，替换成它的上界（通常是 Object）。

  • List<String> 和 List<Integer> 在运行期其实是同一个类。
  • 为什么这么设计？ 为了向后兼容老版本的代码。

三、反射

3.1 什么是反射？

反射是 Java 语言的一种动态机制。它的核心作用是：在程序运行状态中，对于任意一个类，都能够知道这个类的所有属性和方法；对于任意一个对象，都能够调用它的任意方法和属性。

简单来说，常规代码是“先写死类名，再去 new 对象”；而反射允许你在程序跑起来之后，只通过一个字符串（类的全限定名），就能把它加载进来并进行操作。

3.2 为什么要用反射

如果没有反射，所有的代码逻辑在编译期就得全部“写死”，程序的灵活性会大打折扣。

  • 框架的基石：在日常开发中，反射是各种底层框架和中间件的灵魂。以日常构建服务的 SpringBoot 生态为例，当你在代码里打上 @RestController 或 @Autowired 时，Spring 在编译期并不知道你具体写了哪些类。它完全是靠底层在运行时扫描注解，再利用反射机制动态地实例化对象并完成依赖注入的。
  • 动态代理与 AOP：在处理高并发场景下的日志拦截、事务管理时，我们需要在不修改原代码的情况下增强功能，这背后的 JDK 动态代理同样重度依赖反射。

3.3 Class 对象

在反射的世界里，万物皆可被解剖。JVM 在加载每一个类时，都会为它生成一个唯一的 java.lang.Class 对象。这个对象就像是类的“基因图谱”。

反射的主要操作路径如下：

  • 获取图谱 (Class)：通常通过 Class.forName("类的全路径") 或 对象.getClass() 获取。
  • 解析结构：
    • Constructor：获取构造器，用来动态创建对象。
    • Method：获取方法，通过 .invoke() 来执行。
    • Field：获取字段，可以动态读取或修改属性值。

3.4 性能损耗与破坏封装

  • 性能开销：反射代码确实经过了 javac 编译，但它把最核心的类名、方法名、属性名藏在了“字符串”里。这导致编译器变成了瞎子，原本能在编译期完成的安全检查、地址绑定、内联优化，统统被“绕过”，只能把这些沉重的工作推迟到了程序运行的那一刻去动态计算。
  • 破坏封装性：Java 里的 private 关键字在反射面前形同虚设。通过调用 setAccessible(true)（俗称暴力破解），反射可以强行读取和修改类的私有成员变量或调用私有方法。这虽然在框架开发中很强大，但也带来了潜在的安全风险。

四、动态代理

4.1 什么是动态代理？

顾名思义，“代理”就像是明星的经纪人。 你想找明星（目标对象 Target）拍戏，你不能直接联系明星，必须通过经纪人（代理对象 Proxy）。经纪人会在拍戏前帮你“谈片酬”（前置处理），然后让明星去“拍戏”（执行目标方法），拍完后经纪人还会去“收尾款”（后置处理）。

动态代理的核心在于**“动态”：这个经纪人不是你提前写死在代码里的（静态），而是在程序运行期间**，利用反射机制在内存中凭空捏造出来的。

4.2 为什么要用动态代理

在实际的业务开发（比如 SpringBoot 项目）中，我们经常需要在核心业务逻辑（如保存用户）前后，加上各种公共逻辑：

  • 开启/提交数据库事务 @Transactional
  • 记录接口请求日志
  • 校验用户权限

如果不使用代理，你的核心业务代码里就会塞满这些又臭又长的模板代码。动态代理可以将这些“切面逻辑”与“业务源码”完全解耦，在不修改原类一行代码的前提下，偷偷给它加上新功能。

4.3 JDK vs CGLIB

在 Java 世界里，动态代理主要有两种实现方式：

  1. JDK 动态代理
    • 特点：官方原生支持。它要求目标类必须实现至少一个接口。
    • 原理：利用反射生成一个实现了相同接口的代理类。代理类拦截所有方法调用，并把它们转发给一个调用处理器。
    • 局限：如果你的类没有实现任何接口，JDK 代理就束手无策了。
  2. CGLIB 动态代理
    • 特点：第三方字节码技术（Spring 已经内置）。不需要接口也能代理。
    • 原理：在内存中动态生成目标类的一个子类，并重写其中的所有方法（这就是为什么它不能代理被 final 修饰的类或方法）。CGLIB 结结实实地、在字节码层面使用了 Java 的继承机制。

在早期的 Spring 中，官方的策略是：“有接口就用 JDK，没接口才被迫用 CGLIB”。但是到了 SpringBoot 2.x 时代，官方发现 CGLIB 的底层性能优化得太好了，而且不需要开发者强制写接口，开发体验极佳。于是 SpringBoot 2.x 默认把所有的 AOP 代理全部切成了 CGLIB。

4.4 自调用失效问题

当你了解了代理的机制，就能明白日常开发中最容易踩的一个坑：

假设你有一个 UserService 类：

public void a() {
    // 开启事务？
    this.b(); 
}

@Transactional
public void b() {
    // 核心保存逻辑
}

问题： 外部直接调用 a() 时，b() 方法上的事务会生效吗？ 答案：绝对不会生效。 底层逻辑： 因为动态代理是通过代理对象起作用的。外部调用 a() 时走的是代理对象，但进入 a() 方法内部后，this.b() 这里的 this 已经是目标对象本身了（脱离了经纪人）。直接调用原对象的方法，当然不会触发任何增强逻辑。

五、jdk新特性

5.1 Java 8 的函数式革命 (Lambda & Stream)

这是 Java 历史上最重要的一次升级。在 Java 8 之前，Java 是绝对的纯面向对象，连传递一个简单的动作都必须包装在接口（匿名内部类）里。

  • Lambda 表达式 () -> {}：把“行为”当成数据传来传去，代码量瞬间锐减。
  • Stream API：把遍历集合的逻辑从“怎么做（写死 for 循环控制游标）”变成了“做什么（声明式编程）”。你只需要告诉 Stream 帮我 filter() 过滤、map() 转换、最后 collect() 收集，底层怎么遍历的，你不用操心。

5.2 语法糖 (Java 9 - 16)

过去写 Java 最痛苦的就是繁文缛节太多。这几个特性直接让开发体验拉满：

  • 局部变量类型推断 var (Java 10)： 以前你得写：Map<String, List<UserDTO>> map = new HashMap<>(); 现在直接写：var map = new HashMap<String, List<UserDTO>>(); 。

  • 文本块 """ (Java 15)： 以前在 Java 里写一段长 SQL 或者 JSON，每换一行都要加 + 号和 \n，极其恶心。现在直接用三个双引号包起来，所见即所得。

    String json = """
    {
        "name": "Dong",
        "role": "Developer"
    }
    """;

  • 数据载体 Record (Java 16)： 以前为了写一个只包含数据的 DTO（数据传输对象），你要写私有属性、Getter、构造方法、equals、hashCode，或者被迫引入 Lombok 的 @Data 注解。 现在，一行代码搞定一个不可变的类：

    public record UserDTO(String name, int age) {}

5.3 模式匹配(Java 16 - 21)

过去，当你判断一个对象是不是某个类之后，第一件事永远是“强转”。

// 以前的弱智写法
if (obj instanceof User) {
    User u = (User) obj; // 废话！我上面刚查了它是 User，还要我强转！
    System.out.println(u.getName());
}

现在的编译器终于长脑子了，连判断带强转带赋值，一步到位：

// 现在的模式匹配
if (obj instanceof User u) { 
    System.out.println(u.getName()); // 直接用u！
}

在 Java 21 里，这种模式匹配甚至被引入到了 switch 语句中，配合 Record，可以写出极其优雅的代码。

5.4 Java 21 的虚拟线程 (Virtual Threads)

  • 痛点（平台线程 Platform Thread）： 以前我们用的 Thread，在底层是和操作系统的内核线程 1:1 绑定的。非常昂贵！一台服务器顶多开几千个线程就撑爆了。在处理高并发的 Web 请求（比如查数据库、调第三方接口等 I/O 密集型操作）时，线程在等待数据返回的这段时间里，只能傻傻地阻塞在那，白白占着系统资源。
  • 解法（虚拟线程 Virtual Thread）： 虚拟线程是由 JVM 直接管理的极其轻量级的线程。它和操作系统底层真实线程的关系变成了 M:N。你可以毫无压力地创建几百万个虚拟线程！ 最牛的地方在于：当一个虚拟线程遇到 I/O 阻塞（比如等数据库返回）时，JVM 会瞬间把它从底层的真实线程（Carrier Thread）上“拔下来”挂起，让真实线程去跑其他的虚拟线程。等数据库数据回来了，再把它“插回去”继续跑。

虚拟线程的本质： 它是**“同步的编程体验，异步的底层执行”。开发者只管写最简单的阻塞代码，JVM 通过重写底层核心类库，接管了所有 I/O 阻塞点。一旦遇到阻塞，JVM 会自动保存上下文并让出底层真实线程；一旦数据就绪，JVM 又会自动恢复上下文并分配**真实线程继续执行。所有复杂的线程调度、挂起、唤醒，全部由 JVM 在底层默默帮你封装处理了。

六、String

6.1 String 底层数据类型的演进

  • JDK 1.8 及之前： 底层使用 char 类型的数组。
    • 源码：private final char value[]
    • 注：每个 char 占用 2 个字节。
  • JDK 1.9 及以后： 底层全面替换为 byte 类型的数组。
    • 源码：private final byte[] value
    • 进化原因： 官方发现大部分字符串其实只包含英文字母（Latin-1 字符），用 2 个字节的 char 存太浪费内存了。改成 byte[] 加一个编码标志位（coder），能大幅节省堆内存开销。
    • 为了实现紧凑字符串（Compact Strings）优化，Java 9 引入了 coder 标志位。它相当于一个说明书，告诉 JVM 应该怎么去读取当前的 byte[]。 如果 coder 为 0（Latin-1），JVM 就按单字节去读，大幅节省英文字符串的内存； 如果 coder 为 1（UTF-16），JVM 就按双字节去读，保证中文字符的正常显示。 所有的判断和切换都是 JVM 在底层自动完成的，对我们开发者完全透明。

6.2 String 能被继承吗？

绝对不能。

  • 原因： String 类在定义时被 final 关键字修饰。Java 语法规定，final 修饰的类断子绝孙，无法被继承。
  • 源码定义： public final class String implements ...

6.3 String 的“不可变性”是如何实现的

很多面试官会问：“除了类被 final 修饰，还有什么保证了不可变？” 答案分为三个层次：

  1. 类不可继承： 类上的 final 修饰符防止了子类继承并重写（Override）相关方法来搞破坏。
  2. 引用不可变： 保存字符串数据的数组 value 被 final 修饰。这意味着一旦数组被初始化，这个 value 引用就永远不能指向另一个数组。
  3. 内容不可修改（最关键）： value 数组被声明为 private（私有），并且 String 类内部没有任何向外暴露可以修改这个数组内容的公开方法（比如没有任何 setter 方法）。

6.4 String 存储有长度限制吗？

有，且编译期和运行期的限制不同。

  • 编译期限制：65535
    • 底层原因：在编译成 .class 文件时，字符串常量会被放入常量池。JVM 使用 CONSTANT_Utf8_info 结构来表示，这个结构中表示长度的字段只有 2 个字节（16位）。因此，如果你在代码里直接写 String s = "巨长无比的字符串常量...";，最多只能写 65535 个字符，超过就会编译报错。
  • 运行期限制：2^31 - 1
    • 底层原因：运行时，String 内部是用数组存数据的，而 Java 中数组的长度参数 length 是 int 类型的。所以最大长度就是 int 的最大值，即 Integer.MAX_VALUE（大约 20 亿个字符，换算成内存大概是 4GB）。

6.5 为什么要把 String 设计成不可变的

这并非随便设计，而是出于三大核心考量：性能、安全、并发。

  1. 缓存与性能（字符串常量池）
    • 痛点： 字符串是系统里用得最多的对象，疯狂 new 会瞬间撑爆内存。
    • 解法： JVM 在堆区开辟了“字符串常量池”。遇到内容一样的字符串，直接让它们指向池子里的同一个对象，极大节省内存。
    • 不可变的前提： 假设 String s = "abcd"; String s2 = s;，它们指向同一个对象。如果字符串是可变的，你改了 s 的值，s2 莫名其妙就跟着变了，这会引发天下大乱。只有不可变，大家才能安心共享。
  2. 安全性（Security）
    • 场景： 数据库账号密码、网络连接 URL、文件路径，这些极其敏感的信息全是通过 String 传递的。JVM 的类加载器也要靠 String 传类名。
    • 风险： 如果字符串是可变的，黑客就可以在系统校验完 URL 或密码后，悄悄在底层把字符串内容篡改掉。不可变性是 Java 基础安全的基石。
  3. 线程安全（Thread Safety）
    • 优势： 不可变对象天生就是线程安全的。不管多少个线程同时去读它，它的值都在那里，岿然不动。如果某个线程非要改值，对不起，它只能在池子里新建一个字符串，绝对影响不到原有的值和其他线程。

6.6 内存优化核心：字符串常量池

  • 本质： 堆内存中的一个特殊区域，专门用来缓存字符串常量，核心思想是**“复用”**。

  • 代码验证：

    String s1 = "Hello";
String s2 = "Hello";
// .intern() 方法的作用：如果常量池里已经有了"Hello"，就直接返回常量池里的引用；如果没有，就把它放进去。
String s3 = new String("Hello").intern(); 

// 因为指向的都是常量池里同一个地址，所以全部返回 true
System.out.println(s1 == s2); // true
System.out.println(s1 == s3); // true

6.7 String vs StringBuffer vs StringBuilder

这三者的选择是日常开发最容易遇到的场景，直接用表格对比最清晰：

特性StringStringBufferStringBuilder
可变性不可变。每次拼接都会生成新对象。可变。在原对象上修改。可变。在原对象上修改。
线程安全安全（天生不可变）。安全（内部方法加了 synchronized 同步锁）。不安全（无锁，多线程会出问题）。
执行效率最低（频繁拼接会导致内存产生大量废弃对象）。较慢（受同步锁拖累）。最高（无锁轻装上阵）。
适用场景少量字符串操作，或作为常量传递。多线程环境下的字符串大量拼接。单线程环境下的字符串大量拼接（日常开发最常用）。

七、深拷贝、浅拷贝

7.1 核心概念

  • 浅拷贝 (Shallow Copy)： 只复制“外壳”。内部的引用类型变量，依然指向原来的内存地址。
  • 深拷贝 (Deep Copy)： 连外壳带内部数据，彻底复制一份全新的。

7.2 为什么要有这两种区分

因为 Java 里的数据分为两种，它们在内存里的存储方式完全不同：

  1. 基本数据类型（如 int, double）： 直接存的是“值”。不管是深拷贝还是浅拷贝，它们都是直接复制一份值，互不干扰。
  2. 引用数据类型（如 Object, 数组, List）： 存的是“内存地址（指针）”。
    • 如果你用浅拷贝，复制过去的只是“地址”。两个对象的属性最终顺着地址摸过去，修改的是堆内存里的同一个内部对象，极易引发不可预期的 Bug。

7.3 Java 中的实现方案

1. 浅拷贝的官方标准姿势（原生 clone）

Java 所有的类都继承自 Object，而 Object 里面提供了一个 clone() 方法。

  • 做法： 让你自己的类实现 Cloneable 接口，然后重写 clone() 方法。
  • 注意： Java 默认提供的 super.clone() 就是浅拷贝。

2. 深拷贝的三种主流解法

方案具体做法缺点（痛点）优点
方案 A：嵌套调用 clone在重写 clone() 方法时，手动把对象里面嵌套的其他对象也调用一次 clone()。极度恶心。 如果对象里面套了五六层对象，你得每一层都去实现 Cloneable 并重写方法，代码极不优雅。运行速度最快，因为是原生方法。
方案 B：序列化 (Serialization)把对象转换成字节流，然后再从字节流里反序列化出一个新对象。涉及 I/O 流操作，性能开销相对较大。一劳永逸，不管嵌套多少层，只要所有类实现了 Serializable 接口就能搞定。
方案 C：JSON 转换（企业实战最常用）用成熟的库（如 Fastjson, Jackson, Gson），把对象转成 JSON 字符串，再把字符串解析成新对象。性能比原生稍慢，但对于绝大多数日常 Web 业务开发来说，这点性能损耗完全可以忽略不计。代码极简（只需一行代码：JSON.parseObject(JSON.toJSONString(obj), Class)），无需实现任何接口。

八、异常

8.1 异常体系分类

在 Java 中，所有异常的顶级父类是 Throwable，它分为两大派系：

1. Exception 派系（程序自身可以预料并处理的异常）

  • 受检异常 (Checked / Compile-time Exception)：
    • 本质： 编译器强制要求必须处理（try-catch 捕获或 throws 抛出），否则代码直接飘红，无法编译。
    • 常见： IOException（I/O 操作失败）、SQLException（数据库挂了）、ClassNotFoundException。
  • 非受检异常 (Unchecked / Runtime Exception)：
    • 本质： 继承自 RuntimeException。编译器不强制要求处理。这通常是程序员写代码不严谨导致的逻辑漏洞，只要代码足够健壮，完全可以避免。
    • 常见： NullPointerException（臭名昭著的 NPE，空引用）、ArrayIndexOutOfBoundsException（数组越界）、ClassCastException（类型转换错误）。
  • Java规定：
    • 必须捕获的异常，包括Exception及其子类，但不包括RuntimeException及其子类，这种类型的异常称为Checked Exception。
    • 不需要捕获的异常，包括Error及其子类，RuntimeException及其子类。

2. Error 派系（程序无法处理的系统级灾难）

  • 本质： JVM 级别的严重错误。一旦发生，程序自身无能为力，只能乖乖崩溃。它虽然也不需要显式捕获（非受检），但性质和 RuntimeException 完全不同。
  • 常见： OutOfMemoryError（内存撑爆了，OOM）、StackOverflowError（栈溢出，通常是死循环递归导致的）。

8.2 核心对比：throw 与 throws

这两个关键字虽然长得像，但在语法层面和使用场景上完全是两码事。

维度throw (抛出异常动作)throws (声明异常状态)
作用位置方法内部的方法体中。方法签名的尾部。
后面跟什么跟一个异常对象实例（如 new RuntimeException()）。跟一个或多个异常类的名字（用逗号隔开，如 IOException, SQLException）。
执行后果一旦执行到 throw，方法立即终止，它后面的代码绝对不会再执行。只是一个预警声明，告诉调用者：“我这个方法有风险，你调用我时记得处理”。

throw 的神奇之处就在于：它提供了一种跨越层级、强制阻断错误蔓延的机制。如果出了严重 Bug 还用 return 假装没事发生，那后面的业务逻辑拿到错的数据继续跑，后果不堪设想。

8.3 finally 中的代码一定会执行吗？

标准回答起手式： “在绝大多数正常运行和普通的异常捕获场景下，finally 是一定会执行的。它是 Java 提供的一种用来释放资源（如关闭数据库连接、关闭文件流）的兜底机制。但是，有几种极端的底层阻断情况例外。”

导致 finally 彻底失效的 4 种极端情况：

  1. 主动退出 JVM： 在 try 或 catch 块中执行了 System.exit(int status)。这个方法相当于直接把 JVM 这个运行环境给干掉了，皮之不存毛将焉附，finally 自然无法执行。
  2. 强制终止 JVM： 遇到了 Runtime.getRuntime().halt(int status)。它比 System.exit() 更暴力，直接连关闭时的钩子程序（Shutdown Hooks）都不执行，瞬间强制拉闸。
  3. 线程陷入死地： 程序在 try 块里遭遇了死锁（Deadlock）或者死循环，永远卡在那里，代码逻辑根本走不到 finally 这一步。
  4. 物理级别的不可抗力： 比如服务器突然拔掉电源停电、操作系统内核崩溃被强杀（如 Linux 的 kill -9 命令强制杀死进程），或者 JVM 自身出现核心错误崩溃。

九、AIO、BIO、NIO

9.1 同步/异步 与 阻塞/非阻塞 的本质

这两个概念描述的是应用程序（用户空间）与操作系统（内核空间）进行 I/O 交互时的两个不同阶段：

  • 阻塞 (Blocking) vs 非阻塞 (Non-blocking) —— 针对“等待数据就绪时”的线程状态： 当应用程序发起请求，但网卡/磁盘的数据还没准备好时，工作线程是交出 CPU 控制权死等（阻塞），还是立刻返回继续执行后续代码（非阻塞）。
  • 同步 (Synchronous) vs 异步 (Asynchronous) —— 针对“数据就绪后”由谁来搬运数据： 当数据准备好后，是由应用程序的线程自己主动去把数据从内核空间拷贝到应用程序内存（同步），还是由操作系统在底层拷贝好数据，然后直接触发回调函数通知应用程序（异步）。

9.2 BIO、NIO、AIO 总结

BIO（同步阻塞）： 线程发起 I/O 请求后，如果内核没有数据，线程就一直强制休眠等待（阻塞）。等数据终于到了内核缓冲区，该线程还要亲自把数据从内核拷贝到用户空间（同步）。这导致一个连接必须死死霸占一个系统线程，完全无法应对高并发。

NIO（同步非阻塞）：借助多路复用器（Selector），一个线程可以轮询成千上万个连接的状态，没数据时线程立刻返回去查下一个（非阻塞）。但是，只要内核通知某个连接的数据就绪了，该线程依然需要停下轮询，亲自把数据从内核搬运到用户空间（同步）。这是目前主流高并发框架（如 Netty）的底层基石。

AIO（异步非阻塞）： 线程向操作系统发起一个带有缓冲区的读取请求后，直接走人干别的事（非阻塞）。操作系统不仅负责死盯数据是否就绪，还负责把数据直接拷贝到应用程序指定的缓冲区中，全部完工后才触发回调函数通知应用程序（异步）。线程全程被彻底解放，连“搬数据”的活儿都不用干。
