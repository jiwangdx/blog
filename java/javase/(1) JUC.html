一、Java 线程体系中的重要概念

1.1 线程的 6 种状态与生命周期流转

在 Java 中，java.lang.Thread.State 枚举类明确规定了线程的 6 种 状态。

  • NEW (新建状态)：
    • 触发条件： 通过 new Thread() 创建了线程对象，但尚未调用 start() 方法。此时它只是一个普通的 Java 对象。
  • RUNNABLE (可运行状态)：
    • 触发条件： 调用了 start() 方法。
    • 深度解析： 在 Java 虚拟机层面，此状态涵盖了操作系统的“就绪（Ready）”和“运行（Running）”两种状态。抢到了 CPU 时间片就是运行，没抢到就在排队。
  • BLOCKED (阻塞状态)：
    • 进入条件： 线程试图获取一个内部的对象锁（如进入 synchronized 块），但该锁被其他线程持有。
    • 退出条件（回路）： 成功获取到锁后，转回 RUNNABLE 状态。
  • WAITING (无限期等待状态)：
    • 进入条件： 运行中调用了 Object.wait()，此时线程会主动交出 CPU 执行权。
    • 退出条件（回路）： 必须等待其他线程显式地唤醒，如调用 Object.notify() 或 notifyAll()，唤醒后通常需要重新竞争锁，如果没抢到锁会进入 BLOCKED，抢到则进入 RUNNABLE。
  • TIMED_WAITING (计时等待状态)：
    • 进入条件： 调用了带有超时参数的方法，如 Thread.sleep(time)、Object.wait(time)。
    • 退出条件（回路）： 等待时间到达，或者被提前唤醒/中断，随后转回 RUNNABLE 或 BLOCKED（视锁竞争情况而定）。
  • TERMINATED (终止/结束状态)：
    • 触发条件：run() 方法正常执行完毕，或者执行过程中抛出了未捕获的异常导致提前结束。线程一旦终止，无法再被 start() 唤醒。

1.2 线程的创建方式

（1）基础（Inheritance vs. Interface）

继承 Thread 是类级别耦合，任务与线程强绑定，每个new出来的实例都是独立的内存空间（资源不共享），且受 Java 单继承限制，丧失了类扩展性；而实现Runnable是接口级解耦，支持多个线程共享同一个Task实例（资源共享），是面向对象设计的标准实践。

（2）异步通信（Callable/FutureTask）

Callable + FutureTask： 解决的是跨线程结果流转。底层利用 FutureTask 内部的状态机以及 LockSupport 的挂起/唤醒机制，使主线程能以异步非阻塞或阻塞等待的方式感知子线程的执行结果或异常。

（3）线程池（ThreadPoolExecutor）

池化管理： 解决的是系统稳定性与资源复用。通过 ThreadPoolExecutor 维护一组 Worker 线程，将线程的生命周期与任务调度剥离。利用阻塞队列（BlockingQueue）实现流量削峰，通过核心/最大线程数配比规避因频繁创建内核线程（Kernel Thread）导致的系统调用开销及栈溢出（OOM），是生产环境的唯一准则。

1.3 多线程的典型应用场景

（1）异步解耦（提升响应速度）：

场景：用户注册后发送邮件/短信、后台加载大型配置文件或大文件拷贝/迁移。
逻辑：主线程快速返回“操作成功”给用户，耗时的 IO 操作交给后台工作线程默默完成，避免界面卡顿或请求超时。

（2）并发处理（压榨硬件性能）：

场景：爬虫系统抓取多个网页、数据分析时进行分片计算（MapReduce 思想的单机版）。
逻辑：利用多核 CPU 特性，将一个大任务拆分成多个小任务并行执行，最后通过 Future 等机制汇总结果。

（3）高并发服务端模型（网络通信）：

场景：后台服务器处理成千上万的客户端请求、所有的聊天软件（Netty、Tomcat 底层）。
逻辑：每个客户端连接（或请求）分配一个独立线程（或交由核心线程池处理），保证多个用户的请求互不干扰、独立响应。

二、JMM与线程安全

2.1 什么是 JMM（Java 内存模型）

多线程通过共享内存进行通信时，极易出现原子性、可见性、有序性问题。为了解决这些问题，屏蔽掉各种硬件和操作系统的内存访问差异，Java 提出了一套规范——JMM（Java Memory Model）。

JMM 的核心目的：在多线程读写共享数据时，提供统一的规则来保证并发安全。

JMM 的核心架构设计：

  • 主内存（Main Memory）： 线程共享的内存区域，存储所有的共享变量。它是多个线程之间进行数据通信的桥梁。
  • 工作内存（Working Memory）： 每个线程独占的内存区域。用于存储该线程的局部变量、操作结果，以及主内存中共享变量的拷贝副本。
  • 工作模式： 线程对共享变量的所有操作（读取、赋值等）都必须在工作内存中进行，而不能直接读写主内存。操作完成后，再将最新值同步回主内存。

2.2 并发编程带来的线程安全问题

基于主内存与工作内存的交互机制，并发编程中主要面临三大问题：

① 原子性问题 (Atomicity)

  • 痛点： 一个操作或多个操作要么全部执行且不被中断，要么全不执行。例如经典的 i++ 操作，它实际上包含了“读取 - 修改 - 写入”三个步骤，并非原子操作，多线程下必然引发线程安全问题。
  • 破局： 使用 synchronized 关键字（或 ReentrantLock 等锁机制）来保证代码块内的操作具备原子性。

② 可见性问题 (Visibility)

  • 痛点： 线程 A 修改了共享变量的值，但由于修改后的新值只存在于 A 的工作内存中，尚未刷新到主内存；或者线程 B 依然在使用自己工作内存中的旧缓存，导致 B 无法感知到 A 的修改。
  • 破局： 使用 volatile 关键字。
  • 底层原理： 当对 volatile 修饰的变量进行写操作时，JVM 会向处理器发送一条带有 lock 前缀的指令。这会触发底层硬件的缓存一致性协议（如 MESI 协议：Modified Exclusive Shared Or Invalid）。
    • 写回主存： 强制将当前线程工作内存中的缓存行数据立即写回主内存。
    • 缓存失效： 通过 MESI 协议发出信号，使其他 CPU/线程中对应的共享变量缓存行失效。当其他线程需要读取时，由于缓存已失效，必须重新从主内存中加载最新数据，从而保证了可见性。

③ 有序性问题 (Ordering)

  • 痛点： 为了提升执行效率，编译器和处理器经常会对指令进行重排序（包括编译器重排序和指令级并行重排序）。在单线程下重排序不会改变最终结果，但在多线程下可能会导致致命问题。
  • 破局： 使用 volatile 关键字禁止指令重排序。
  • 底层原理（内存屏障）：volatile 通过在特定位置插入内存屏障（Memory Barrier，一种 CPU 指令）来实现有序性。编译器在生成字节码时，会在指令序列中插入屏障来阻止特定类型的处理器重排。

三、线程池

3.1 为什么要用线程池

“使用线程池的核心目的可以用四个字概括：‘降耗提速’。

首先，它通过线程复用，规避了频繁创建和销毁线程带来的内核态切换开销。 其次，它提高了响应速度，任务到达时无需等待线程创建即可立即执行。 最后，它提供了强大的资源管理能力，防止因为无限制地开启线程导致系统资源（如内存、CPU）耗尽。”

3.2 线程池的执行流程

提交一个任务时：

  1. 先判断核心线程数（corePoolSize）是否已满，不满则直接创建执行；
  2. 满了则进入阻塞队列（workQueue）缓冲；
  3. 如果队列也满了，再看最大线程数（maximumPoolSize），未满则创建临时线程增援；
  4. 如果人手和仓库都到了极限，就会触发拒绝策略（handler）。

此外，通过线程工厂（threadFactory）我们可以给线程起名方便排查，而存活时间（keepAliveTime/unit）则负责在任务低谷期回收临时线程，保持系统弹性，即超时销毁。

3.3 拒绝策略的选择

  • 绝对不能丢任务， CallerRunsPolicy（调用者运行）。它让提交任务的线程（比如处理请求的tomcat线程）自己去干活，既保证任务不丢，又能起到‘反向压力’的作用，逼迫提交者降速。
  • 如果追求实时性（如最新的股价显示），我会选 DiscardOldestPolicy，‘喜新厌旧’，把最老的任务踢出去，给新数据腾位置。
  • 如果是无关紧要的辅助逻辑（如日志记录），我会选 DiscardPolicy 默默丢弃。

虽然CallerRunsPolicy不丢任务，但如果你的任务执行特别慢，会导致大量 Tomcat 线程被占用，最后用户会发现整个网页都打不开了（因为没有空闲的 Tomcat 线程来接收新连接了）。所以，这种策略只适合那些“任务虽多但单次执行不算太慢”的场景。

从操作系统内核层面来看，Tomcat 线程和自定义的业务线程是完全等价的，都是普通的内核级线程，享有相同的 CPU 调度权重，执行速度没有区别。

我们在架构上将它们分开，并不是因为谁执行得更快，而是为了做资源隔离。

Tomcat 线程池专门负责处理极其快速的网络 I/O 和请求分发；而自定义业务线程池专门负责处理耗时的核心业务逻辑（如数据库读写）。这种物理上的隔离保证了，即使后端业务出现阻塞或慢 SQL，耗尽了业务线程，Tomcat 依然有充足的线程来接收新的用户请求并快速返回降级提示，从而避免了整个 Web 服务的‘雪崩’。

3.4 线程池参数的确定

（1）一个理论基石（温伯格公式）

一句话：不让 CPU 闲着。

  • 本质： 等待时间越多（IO多），线程数就该越多，用来填补 CPU 的空闲。

（2）两个经典场景

场景类型推荐公式核心逻辑
CPU 密集型N + 1CPU 几乎没等待。多出来的 1是为了在偶发页缺失或中断时，保持 CPU 满载。
I/O 密集型2N或更高大部分时间在等网络/磁盘。增加线程是为了让 CPU 在 A 等待时能去干 B 的活。

（3）实战调优

初始定值： 根据公式和业务是 IO 型还是计算型，先设一个基准值（如 2N）。

压测迭代： 使用监控工具（如 Prometheus）观察 CPU 利用率 和 吞吐量。

如果 CPU 没满，但队列满了，说明线程数太少。
如果 CPU 爆了，但吞吐量上不去，说明线程数太多，上下文切换（Context Switch）损耗太严重。

动态观测： 上线后根据实际情况进行动态微调。

上下文切换就是 CPU 为线程执行“存档”与“读档”的过程，由于涉及程序计数器和寄存器状态在内核态的保存与恢复，开销极大；在工程上应通过合理配置线程数、使用 CAS 无锁编程及缩小锁粒度来减少线程阻塞，从而降低切换频率以压榨 CPU 性能。

3.5 上线后关注线程池的哪些指标

活跃度：看活跃线程是否接近最大值，判断是否需要扩容；

队列余量：这是防止内存溢出（OOM）的第一道防线；

任务耗时：包括执行时间和排队时间，用来定位代码性能瓶颈；

拒绝次数：这是最直观的异常指标，直接反映了系统是否已经过载。

如果发现指标异常，进行动态调整。Java 的 ThreadPoolExecutor 提供了 setCorePoolSize 和 setMaximumPoolSize 的方法。我们可以不重启服务，直接在线上动态调参来临时救火。

3.6 为什么不推荐使用jdk的Executors创建线程池？

Executors 是一个工厂类，它提供了用于创建不同类型线程池的静态方法

线程池不允许使用 Executors 去创建，而是通过ThreadPoolExecutor的方式，规避资源耗尽的风险。

Executors有两个无限：队列无限大（Integer.MAX_VALUE）；线程无限多（Integer.MAX_VALUE）。

四、ThreadLocal 深度解析与避坑指南

4.1 核心定位与本质

ThreadLocal 提供了一种线程级别的数据隔离机制。它为每个使用该变量的线程提供了一个独立的变量副本，使得每个线程都可以独立地、安全地操作自己的数据，而不会与其他线程产生冲突。

核心作用：

数据传递： 避免在复杂的业务链路（多层方法调用）中频繁传递参数。

线程隔离： 解决多线程并发下的数据安全问题。

核心伪代码骨架：

// 1. 定义全局的“口袋”
static ThreadLocal<String> userHolder = new ThreadLocal<>();

// 模拟 Tomcat 线程处理一次 HTTP 请求的完整生命周期
public void handleHttpRequest() {
    try {
        // --- [1. 网关/拦截器层] ---
        String userId = "User-9527"; // 解析 Token 获得身份
        userHolder.set(userId);      // 【存】塞进当前线程，透传开始

        // --- [2. 深层业务 Service 层] ---
        // 核心魔法：方法不用传参，直接跨层掏口袋
        String currentUser = userHolder.get(); // 【取】拿到 "User-9527"
        // 执行业务逻辑：createOrder(currentUser)...

    } finally {
        // --- [3. 拦截器出口] ---
        userHolder.remove();         // 【删】铁律：用完必须倒垃圾！防泄漏与串用
    }
}

一句话速记核心骨架： 声明一个 static ThreadLocal，在入口处 try { set() }，在业务中途随时 get()，在出口处 finally { remove() }。

典型的ThreadLocal工具类：

public class UserContextHolder {
    // 静态的 ThreadLocal 变量，保证全局可用，但数据在各线程间隔离
    private static final ThreadLocal<String> userHolder = new ThreadLocal<>();

    // 存入当前用户的 ID
    public static void set(String userId) {
        userHolder.set(userId);
    }

    // 获取当前用户的 ID
    public static String get() {
        return userHolder.get();
    }

    // 清理当前用户的 ID（极其重要！）
    public static void remove() {
        userHolder.remove();
    }
}

思考：为什么get直接就拿到了当前对应的value

// ThreadLocal.java 内部的 get() 源码核心逻辑
public T get() {
    // 1. 获取当前正在执行代码的线程（拿到当前员工）
    Thread t = Thread.currentThread();
    
    // 2. 获取这个线程专属的 ThreadLocalMap（摸他身上的口袋）
    ThreadLocalMap map = getMap(t);
    
    if (map != null) {
        // 3. 【核心魔法就在这！】 
        // 这里的 this 是谁？谁调用的 get，this 就是谁！
        // 因为是你调用的 userHolder.get()，所以这里的 this 就是 userHolder 对象本身！
        ThreadLocalMap.Entry e = map.getEntry(this); 
        
        if (e != null) {
            return (T) e.value;
        }
    }
    return setInitialValue();
}

4.2 典型业务场景

在实际企业级开发中，ThreadLocal 的应用极为广泛：

  • 用户身份/鉴权透传： 在网关或拦截器中校验 Token 后，将用户信息（如 UserID）存入 ThreadLocal，后续所有的业务 Service 层直接获取，做到“一次拦截，全链路通行”。
  • 非线程安全对象的封装： 如 SimpleDateFormat，通过 ThreadLocal 为每个线程创建一个独立实例，完美避开并发格式化报错。
  • 全链路日志追踪（MDC）： 存储 traceId。在微服务打印日志时，自动从 ThreadLocal 取出 traceId 拼接到日志中，方便日后的日志采集合并与排障。
  • 数据库 Session管理： ORM 框架（如 MyBatis、Spring 事务管理）通过 ThreadLocal 确保同一个线程内的所有数据库操作使用的是同一个 Session。

4.3 底层架构与实现原理

ThreadLocal的本质是将数据存放在当前线程对象中，而不是ThreadLocal本身。

（1）存储结构：Thread -> ThreadLocalMap -> Entry

  • 每个 Thread 对象内部都维护着一个 ThreadLocalMap 成员变量。
  • ThreadLocalMap 底层是一个 Entry[] 数组。
  • Entry 的结构：**Key 是 ThreadLocal 对象本身，Value 是你要存储的具体数据。
  • 结论：一个线程可以有多个 ThreadLocal（存用户、存事务、存日志），它们都会存入当前线程的这个 Entry[] 数组中。

（2）Hash 散列与冲突解决

  • 斐波那契散列：每个ThreadLocal在创建时都会分配一个独特的threadLocalHashCode，以此计算在Entry[]数组中的索引，保证极致的散列均匀。
  • 线性探测法（Linear Probing）：与HashMap遇到冲突采用链表不同，ThreadLocalMap没有链表结构。当发生 Hash 冲突时，它会向后顺序遍历找下一个空位（i + 1）放入。

4.4 内存泄漏问题

ThreadLocal 面试必考点在于内存泄漏，而彻底理解它的关键在于搞懂 Entry 的引用设计。

// ThreadLocalMap 的 Entry 源码：Key 继承了弱引用
static class Entry extends WeakReference<ThreadLocal<?>> {
    Object value; // Value 是强引用
    Entry(ThreadLocal<?> k, Object v) {
        super(k);
        value = v;
    }
}

（1）为什么会发生内存泄漏？

在**线程池（Thread Pool）**场景下，核心线程是常驻且复用的，不会被销毁。 如果我们将数据放进了 ThreadLocal，此时引用链为：Thread -> ThreadLocalMap -> Entry -> Value。 因为 Value 是强引用，只要线程不死，这个 Value 就永远存在于内存中无法被 GC 回收，长期累积直接导致 OOM。

（2）既然危险，为什么 JDK 还要把 Key 设计成弱引用？

这其实是 JDK 的一层主动防御（兜底）机制。

  • 如果 Key 也是强引用： 只要线程池的线程在，ThreadLocalMap 就会死死拉住 ThreadLocal 对象不让它销毁。即便外部业务代码已经不再使用该 ThreadLocal 了，它也会引发严重泄漏。
  • 设计为弱引用后： 当外部对 ThreadLocal 的强引用断开，下一次 GC 时，这个 ThreadLocal（Key）就会被回收掉。此时 Entry 中就会出现 Key == null 的情况（称为 脏数据 Stale Entry）。
  • 启发式清理**：**ThreadLocal 在后续被调用 get()、set()、remove() 方法时，底层会自动扫描并触发 expungeStaleEntry() 方法，将这些 Key == null 的 Entry 的 Value 也置为 null，帮助 GC 回收。

4.5 上下文污染问题

除了内存泄漏，在 Web 容器（如 Tomcat）使用线程池处理请求时，不清理 ThreadLocal 会导致极其严重的安全越权事故（数据串用）。

场景重现：

  1. 线程 A 被分配处理【用户张三】的请求，将张三的信息存入 ThreadLocal。
  2. 请求结束，代码有缺陷，**未执行 **remove()。
  3. 线程 A 被归还给 Tomcat 线程池。
  4. 接着，线程 A 又被分配去处理【用户李四】的请求。
  5. 此时如果代码逻辑直接去 get()，拿到的竟然是张三的信息！导致李四看到了张三的数据，产生越权 bug。

4.6 最佳工程实践

为了彻底规避内存泄漏和上下文污染，生产环境使用 ThreadLocal 必须遵守唯一铁律：

**必须在 **try-finally 块的 **finally 中调用 **remove() 方法清理。

// 标准的 Web 拦截器使用范例
public class UserInterceptor implements HandlerInterceptor {

    @Override
    public boolean preHandle(...) {
        // 请求进入，解析 Token 并存入 ThreadLocal
        UserInfo user = parseToken(request);
        UserContextHolder.set(user);
        return true;
    }

    @Override
    public void afterCompletion(...) {
        // 请求结束，无论是正常返回还是抛出异常，必须强制清空！
        UserContextHolder.remove();
    }
}

五、Synchronized

5.1 Mark Word

Java 中每个对象在内存中的头部结构里，都有一个极其关键的区域——Mark Word（标记字段）。它是一个动态的数据结构，存储着对象的哈希码、GC 分代年龄等，最重要的是它记录了当前的锁状态信息。

5.2 Synchronized 的分层实现原理

synchronized 的同步机制是自上而下分层实现的：

  1. 语言层面（Syntax）： 表现为 synchronized 关键字，是最基础的同步原语。
  2. 字节码层面（Bytecode）： 编译为 monitorenter 和 monitorexit 两条指令，圈定临界区，通知 JVM 加锁与解锁。
  3. JVM 逻辑层面（Monitor 管程）：
    • 每个锁对象天生关联一个 Monitor。
    • Owner： 指向当前持有锁的线程（monitorenter 即尝试成为 Owner）。
    • EntryList： 等待锁的阻塞线程队列（Owner 被占时，其他线程进入此队列；monitorexit 释放锁时，唤醒此队列的线程重新竞争）。
  4. 操作系统层面（OS Kernel）： 最终依赖操作系统的 Mutex Lock（互斥量）实现。线程的阻塞和唤醒需要从用户态切换到内核态，开销极其昂贵。

5.3 锁升级 (JDK 1.6+)

为了打破频繁切换“内核态”带来的性能瓶颈，JDK 1.6 引入了锁升级机制，核心设计思路是：为了适配不同竞争强度下的场景，动态调整加锁的成本。

锁的状态会根据竞争情况，沿着 无锁 -> 偏向锁 -> 轻量级锁 -> 重量级锁 单向膨胀：

  • 阶段一：无锁 -> 偏向锁 (单线程的 VIP 通道)
    • 场景： 没有任何锁竞争，且同一个线程需要反复重入该代码块。
    • 机制： 线程首次进入时，JVM 将锁标记为偏向锁，并在锁对象的 Mark Word 中记录下该线程的 ID。如果该线程再次重入，只需对比 ID 是否一致，一致则直接放行，几乎零开销。
  • 阶段二：偏向锁 -> 轻量级锁 (少量线程极短时间的竞争)
    • 场景： 突然有其他线程来请求锁，打破了单线程的 VIP 状态。
    • 机制： 偏向锁被撤销。竞争线程会在自己的线程栈中开辟一个“锁记录（Lock Record）”空间，把对象的 Mark Word 复制过去。随后，多个线程通过 CAS (Compare-And-Swap) 操作，尝试将对象头部的 Mark Word 更新为指向自己 Lock Record 的指针。
    • 优势（自旋锁）：竞争失败的线程不会立刻陷入阻塞，而是原地执行“空循环（自旋）”等一会儿。这完全在用户态完成，避免了内核切换的昂贵代价。
  • 阶段三：轻量级锁 -> 重量级锁 (高并发的惨烈厮杀)
    • 场景： 短时间内有大量线程竞争，或者拿到锁的线程执行太慢，导致其他线程自旋太久。
    • 机制： 发生锁膨胀，升级为重量级锁。此时，所有竞争失败的线程彻底放弃自旋，全部进入 JVM 层面的 EntryList 陷入阻塞状态，交出 CPU 执行权，直到锁释放被操作系统内核唤醒。

六、ReentrantLock 与 AQS 框架

synchronized 虽然经过了锁升级优化，但在极度复杂的并发场景下（如需要超时响应、中断等待、多条件唤醒），它显得过于死板。J.U.C 提供了基于 AQS 框架的显式锁机制来破局。

6.1 J.U.C 的心脏：AQS (AbstractQueuedSynchronizer)

AQS 是整个 Java 并发包的底层基石，不管是 ReentrantLock 还是 CountDownLatch、Semaphore，底层全是由它驱动的。它本质上是一个依赖状态位和队列的同步器模板。

AQS 的三大核心组件：

  1. volatile int state (共享资源状态位)：
    • AQS 的灵魂。通过底层的 CAS 机制 保证对 state 修改的原子性。
    • 多态语义： 在不同子类中代表不同含义。
      • 在 ReentrantLock 中：0 代表无锁，1 代表被加锁，>1 代表被同一个线程重入加锁。
      • 在 CountDownLatch 中：代表还剩多少个计数任务。
  2. FIFO 双向等待队列：
    • 当线程通过 CAS 抢占 state 失败时，AQS 会将该线程包装成一个 Node 节点，加入到双向链表的尾部，并通过 LockSupport.park() 将线程挂起阻塞。
    • 当持有锁的线程释放锁时，会通过 LockSupport.unpark() 唤醒队列头部的下一个节点。
    • AQS 本质上就是一个依赖双向链表的排队休眠机制：抢不到锁的线程乖乖入队挂起睡觉（不白耗 CPU），等持有锁的线程释放资源走人时，负责顺着指针精确唤醒身后的下一个线程起来接班。
  3. 资源共享模式：
    • 独占模式 (Exclusive)： 只有一个线程能修改 state 成功（如 ReentrantLock）。
    • 共享模式 (Shared)： 多个线程可以同时成功修改 state（如 Semaphore、CountDownLatch）。

6.2 ReentrantLock 的高阶特性

相比于 synchronized 只能死等，ReentrantLock 提供了极高的灵活性：

  • 可中断响应： 使用 lockInterruptibly() 加锁时，如果线程在等待队列中被 interrupt，它会立刻抛出异常并退出等待，防止死锁。
  • 超时尝试： 使用 tryLock(long time, TimeUnit unit)，如果在指定时间内拿不到锁，直接返回 false，业务可以立刻去做降级处理。
  • 多条件等待 (Condition)：synchronized 只能关联一个 wait()/notify() 队列，而 ReentrantLock 可以 newCondition() 绑定多个等待队列，实现精准唤醒。

6.3 公平锁与非公平锁

ReentrantLock默认采用非公平锁，因为非公平锁的吞吐量极高。

  • 非公平锁 (Non-fair)：允许“强行插队”。
    • 机制： 当新线程来请求锁时，根本不管 AQS 队列里有没有人排队，直接一上来就执行 CAS 尝试把 state 从 0 改为 1。如果碰巧上一个线程刚释放锁，它就插队抢夺成功；抢失败了，才乖乖去队尾排队。
    • 优势： 极大地减少了线程唤醒（从内核态切换回用户态）的开销。
  • 公平锁 (Fair)：绝对的 FIFO 先来后到。
    • 机制： 源码底层多了一个判断：hasQueuedPredecessors()。新线程来请求锁时，先看看 AQS 队列里有没有前驱节点排队。哪怕此时锁刚好被释放，只要自己不是排在第一位，也必须去队尾排队。
    • 缺点： 频繁触发线程挂起和唤醒，吞吐量远低于非公平锁。

七、无锁并发：CAS、Unsafe 与原子类

为了避免传统加锁（synchronized）带来的线程阻塞挂起，Java 采用 CAS（乐观思想）来保证并发安全；而为了实现 CAS 的绝对原子性，Java 通过 Unsafe 类直接调用操作系统的硬件指令；为了让开发者安全地使用这个危险特性，JDK 又将其封装成了 原子类 供日常开发调用。

7.1 CAS (比较并交换)

更新数据时，先看看数据有没有被别人偷偷改过。没改过就更新，改过了就重新读取再试一次（自旋）。

  • V (内存实际值)： 变量现在真实的数值。
  • E (我的预期值)： 我刚才读取到的旧数值。
  • N (计算的新值)： 我想把它更新成的目标数值。
  • 动作： 只有当 V == E 时，才把数值更新为 N。

7.2 Unsafe 类

  • 作用： JVM 留给底层的“特权后门”。Java 代码本身无法直接操控硬件，必须通过 Unsafe 类调用底层的 C/C++ 代码，最终生成一条 CPU 的硬件锁总线指令（cmpxchg），以此来保证 CAS 动作绝对不被打断。
  • 代价： 极度危险，它可以像 C 语言一样直接操作内存地址，用错会导致 JVM 崩溃或内存泄漏。

7.3 原子类 (Atomic)

因为 Unsafe 不能随便用，JDK 在 java.util.concurrent.atomic 包下将其进行了安全封装（如 AtomicInteger）。当我们调用 incrementAndGet() 时，底层就是 CAS + Unsafe + volatile 的完美结合。

八、悲观锁 vs 乐观锁

前言核心： 悲观锁与乐观锁并不是某一种具体的锁机制（不是具体代码），而是解决并发冲突的两种思想流派。无论是在 Java 内存中（JVM），还是在关系型数据库中（MySQL），都遵循这两种哲学。

8.1 悲观锁 (Pessimistic Locking)：先上锁，后办事

  • 核心思想： 极度缺乏安全感。认为只要自己不加锁，别人一定会来搞破坏。因此在对数据进行读写前，必须强行加排他锁。
  • 技术落地：
    • Java 层面：synchronized 关键字、ReentrantLock。
    • 数据库层面： 行锁/表锁，如 SELECT * FROM table WHERE id = 1 FOR UPDATE;
  • 应用场景（写多读少）：
    • 极度要求数据绝对安全、且并发写冲突极高的场景。
    • 例如： 银行核心账务系统（资金划拨）、电商大促时的超热点商品库存绝对锁定。
  • 致命痛点： 把多线程的并行执行强行变成了串行化。未抢到锁的线程全部陷入阻塞排队，导致系统吞吐量极速下降；且在复杂业务中极易引发死锁（Deadlock）。

8.2 乐观锁 (Optimistic Locking)：先办事，后检查

  • 核心思想： 极度自信。认为系统大部分时间是安全的，别人不会来改数据。所以操作前不加锁，只在最后提交数据的那一刻，回头看一眼数据有没有被别人动过。没动过就提交，动过就报错或重试。
  • 技术落地：
    • Java 层面： CAS 机制（Compare And Swap）+ 原子类（AtomicInteger 等）。
    • 数据库层面： 版本号机制（Version 字段）或时间戳机制。
  • 应用场景（读多写少）：
    • 冲突频率较低，且对系统并发吞吐量要求极高的场景。
    • 例如： 维基百科词条编辑（谁先提交算谁的）、个人主页信息修改、普通商品的库存扣减。
  • 致命痛点： 如果评估失误，在冲突极高的场景下用了乐观锁，会导致大量线程在最后提交时失败，陷入无限的“失败 -> 重试 -> 失败”的自旋中，疯狂燃烧 CPU 资源。
