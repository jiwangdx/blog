一、 MySQL 架构设计与执行流转

1.1 一条 SQL 查询语句的执行过程

一条 SQL 从发送到返回结果，在 MySQL 内部会经历完整的流水线处理：

（1）连接器：建立连接，校验权限。

（2）查询缓存（MySQL 8.0 已废弃）：命中缓存直接返回，未命中则继续。废弃原因是表一更新缓存就失效，性价比极低。

（3）分析器：词法分析（识别出关键字 select、表名、字段名）和语法分析（检查 SQL 语句是否符合语法规则）。

（4）优化器：核心大脑。决定使用哪个索引，决定表连接的顺序，生成最终的执行计划。

（5）执行器：调用存储引擎的 API 接口，逐行读取数据进行过滤，最终返回结果集。

1.2 查询关键字的底层执行顺序

程序员写 SQL 的顺序和 MySQL 执行的顺序是完全不同的！

（1）执行顺序流：FROM -> JOIN -> ON -> WHERE -> GROUP BY -> HAVING -> SELECT -> ORDER BY -> LIMIT。

（2）核心启示：因为 WHERE 在 SELECT 之前执行，所以 WHERE 中不能使用 SELECT 阶段定义的别名；而 ORDER BY 在 SELECT 之后，所以可以放心使用别名排序。

1.3 核心存储引擎对比：InnoDB vs MyISAM

（1）InnoDB（MySQL 5.5 后的默认引擎）

  • 特性：支持事务、支持行级锁（并发性能极高）、支持外键。
  • 结构：数据和索引绑定在一起存储（聚簇索引）。
  • 适用场景：绝大多数的互联网业务、高并发读写、对数据一致性要求高的场景。

（2）MyISAM

  • 特性：不支持事务，只支持表级锁，不支持外键，但会额外保存整张表的总行数（count(*) 极快）。
  • 结构：数据和索引分开放置（非聚簇索引）。
  • 适用场景：历史归档数据、纯读无写的报表系统。

二、 索引底层原理与实战调优

2.1 B+树索引机制

（1）为什么选择 B+ 树而不是二叉树、红黑树或 Hash

  • 相比二叉树/红黑树：B+ 树是“矮胖型”的，每个节点可以容纳更多的数据，大大降低了树的高度。数据库数据存在磁盘上，树越矮，磁盘 I/O 次数越少，查询越快。
  • 相比 Hash：Hash 只能做等值查询，而 B+ 树的叶子节点之间用双向链表相连，**完美支持范围查询（Range Query）**和排序。

（2）B+ 树的结构特征：非叶子节点只存索引键，不存真实数据；所有真实数据只保存在底层的叶子节点中。

2.2 聚簇索引与非聚簇索引（二级索引）

（1）聚簇索引 (Clustered Index)

  • 原理：“索引即数据，数据即索引”。B+ 树的叶子节点直接挂载了整行完整的数据。通常主键就是聚簇索引。

（2）非聚簇索引 (Secondary Index)

  • 原理：基于普通字段（如 name）建立的索引。它的 B+ 树叶子节点中只存了索引字段的值和对应的主键 ID，没有整行数据。

（3）回表查询

  • 当通过普通索引查到主键 ID 后，如果查询还需要其他字段的值，就必须拿着这个主键 ID 再去“聚簇索引”树上查一遍拿完整数据，这个动作叫**“回表”**。

2.3 索引失效的常见情况

（1）违反最左前缀法则：对于联合索引 (a, b, c)，查询条件必须从最左边的 a 开始连续匹配，如果直接用 where b=1 and c=2，索引将完全失效。

（2）隐式类型转换：字符串类型的字段，查询时不加单引号（如 where phone = 13800000000），MySQL 底层会截断转型，导致索引失效。

（3）在索引列上做运算：对索引字段使用了函数或数学运算（如 where DATE(create_time) = '2023-01-01'），会导致引擎放弃索引进行全表扫描。

（4）模糊查询：以 % 开头的 LIKE 查询（如 like '%张三'）一定会导致索引失效。

2.4 慢sql的定位与优化

（1）找到慢SQL

开启慢查询日志（Slow Query Log）

  • 这是数据库层面最直接的手段。记录的是纯粹的数据库内部执行时间。它包含了 SQL 的解析、优化、锁等待、引擎执行等时间，但不包含网络传输时间。

    • 在 MySQL 中，可以配置 slow_query_log = 1 并设置 long_query_time（比如 1 秒或 2 秒）。
    • 运行一段时间后，MySQL 会将所有执行时间超过该阈值的 SQL 写入日志文件。
    • 工具辅助： 可以使用 mysqldumpslow 或 pt-query-digest 等工具对慢查询日志进行聚合分析，找出执行频率最高、耗时最长的“Top N” SQL。

应用层监控与链路追踪

  • 记录的是JDBC调用的整体时间。这包含了：应用发送SQL到数据库的网络耗时 + 数据库执行耗时 + 数据库把结果集返回给应用的网络耗时。

    • 日志拦截： 在代码层面，可以通过 Spring AOP 统一记录 SQL 的执行时间，超过阈值则打印 Warn 日志。
    • APM 工具： 在生产环境中，通常会接入 SkyWalking等监控工具。这些工具可以提供接口级别的响应时间分布，并直接下钻查看到底是哪一条 JDBC 调用的 SQL 拖慢了接口。接口级别指的是会清清楚楚地列出你们系统里的每一个具体业务功能

实时排查（应急处理）

    • 如果线上突然发生卡顿或 CPU 飙升，可以直接在数据库执行 SHOW PROCESSLIST;。这会列出当前正在执行的线程，重点关注 Time（执行时长）较高且 State 为 Sending data的会话，直接揪出正在作恶的 SQL。

（2）使用 EXPLAIN 分析

拿到具体的慢 SQL 后，绝对不要急着改代码，第一步永远是拿 EXPLAIN 命令去跑一下这条 SQL，看看数据库的执行计划。

重点关注 EXPLAIN 返回结果中的以下几个字段：

  • type（连接类型）： 代表了查询的效率，性能从坏到好依次为：

    • ALL：全表扫描（最糟糕，必须优化，除非表极小）。
    • index：全索引扫描。
    • range：索引范围扫描（如 BETWEEN、>、<，一般要求优化到此级别）。
    • ref：非唯一性索引扫描。
    • eq_ref / const：主键或唯一索引扫描（性能最佳）。

  • possible_keys & key： 前者是可能用到的索引，后者是实际用到的索引。如果 key 为 NULL，说明没走索引。

  • rows： MySQL 预估需要扫描的行数。这个值越大，通常代表查询越慢。

  • Extra（额外信息）： 包含非常重要的性能提示：

    • 高危警告：Using filesort（说明进行了外部排序，通常是因为 ORDER BY 没走索引）、Using temporary（使用了临时表，常见于 GROUP BY 或多表 JOIN）。
    • 优秀标志：Using index（触发了覆盖索引，直接从索引树中拿到了所有数据，没有进行回表操作）。

（3）解决与优化

根据 EXPLAIN 的结果，绝大部分慢 SQL 都可以通过以下几个维度来解决：

  • 创建缺失的索引： 如果 type 是 ALL，检查 WHERE 条件和 ORDER BY、GROUP BY 的字段，为其建立合适的索引。

  • 遵循最左前缀法则： 对于联合索引 (a, b, c)，查询条件必须包含 a，才能走到索引。如果是 WHERE b=1 AND c=2 是不会走联合索引的。

  • 利用覆盖索引防回表： 如果你只需要查 id 和 name，且刚好有 (name) 的索引，那么就直接 SELECT id, name，不要用 SELECT *。这样数据库扫完辅助索引树就能返回结果，不需要回表。

  • 避免索引失效的“骚操作”：

    • 不要在索引列上做运算或使用函数（例如 WHERE YEAR(create_time) = 2026 会导致全表扫描，应改为 WHERE create_time >= '2026-01-01' AND ...）。
    • 避免隐式类型转换（比如字段是 VARCHAR，但查询写了 WHERE phone = 13800000000，数字不加引号会导致索引失效）。
    • 慎用左模糊或全模糊查询（LIKE '%keyword' 不走索引，只有 LIKE 'keyword%' 才走）。

  • 优化深度分页（Deep Paging）：

    • 像 LIMIT 100000, 10 这种写法，数据库会扫描 100010 行然后丢弃前 10 万行，非常慢。
    • 解法： 基于游标（上一页最后一条的主键作为下一页的起始条件 WHERE id > last_id LIMIT 10）。

  • 用 **UNION ALL** 替代 **UNION**： 如果确定两部分结果集没有重复数据，或者业务不关心重复，务必使用 UNION ALL，因为它去掉了耗时的去重动作。

如果 SQL 已经写到了极致，索引也建得很完美了，但依然慢，那就需要跳出数据库层面来看问题：

  • 业务裁剪： 这个极其复杂的 JOIN 操作，真的是一次请求必须返回的吗？能否在应用层拆分成两三次简单的单表查询，然后在内存中用 Java 代码去做数据拼装？
  • 引入缓存： 对于读多写少、时效性要求不高的数据，前面挡一层 Redis，直接拦截掉绝大部分数据库打压。
  • 分库分表 / 读写分离： 单表数据量是否已经过大（例如超过千万级）？可以考虑将历史冷数据归档，或者实施分库分表方案。将复杂的分析类统计查询路由到只读从库，避免影响主库的写入和在线交易。

在 SQL 语句前加 EXPLAIN 关键字，可以查看 MySQL 打算如何执行这条语句。重点关注三个字段：

（1）type (连接类型)：衡量查询效率的关键。性能从好到坏依次是：system > const > eq_ref > ref > range > index > ALL。如果是 ALL（全表扫描），必须优化。

（2）key：实际决定使用的索引。如果为 NULL，说明没走索引。

（3）Extra：额外信息。出现 Using filesort（外部排序）或 Using temporary（使用临时表）时性能极差，亟需优化；出现 Using index 说明实现了覆盖索引，性能极佳。

2.5 为什么 MySQL 采用 B+ 树作为索引？

MySQL 的索引本质是为了提高查询效率的数据结构。
哈希表虽然查询快，但不支持范围查询；
二叉树和红黑树分叉度低，数据量大时树高过高，磁盘 I/O 次数多；
B 树是多路平衡树，但节点中既存索引又存数据，范围查询效率一般；
B+ 树非叶子节点只存索引，叶子节点顺序存储并通过链表连接，树高更低、范围查询更高效，因此 MySQL 选择 B+ 树作为索引结构。

三、 事务并发控制与 MVCC

3.1 mysql 隔离级别

（1）事务隔离级别概述

事务隔离级别定义了多个事务并发执行时数据的可见性规则，核心目标是平衡数据一致性与并发性能。不同级别通过锁机制和 MVCC（多版本并发控制）实现。

隔离级别脏读不可重复读幻读实现机制
读未提交（RU）❌❌❌无锁或共享锁
读已提交（RC）✅❌❌MVCC每次生成新ReadView
可重复读（RR）★✅✅⚠️MVCC复用ReadView + 间隙锁
串行化（Serial）✅✅✅所有操作加锁（表锁/行锁）

💡 说明：

  • ✅：可避免该问题；❌：可能出现；⚠️：理论可能，但InnoDB通过间隙锁解决。
  • MySQL 默认级别为 RR，因其在保证一致性的同时兼顾性能。

**脏读（Dirty Read）：**你读到了别人还没点“保存”的临时数据，一旦对方最后点了“取消撤回”，你拿去用的就是根本不存在的假数据。

不可重复读（Non-Repeatable Read）：你死死盯着的同一条特定数据，在你还没办完事的时候被别人成功修改或删除了，导致你前后两次看的结果完全变了。 12306

幻读（Phantom Read）：你查询某一片区域的数据，别人趁机在里面偷偷新增了一条新记录，导致你再次核对时，像见鬼一样凭空多出了一个原本没有的东西。

（2）读未提交（RU）

作为隔离级别体系中最底层的一环，RU 采取了几乎放弃并发读写控制的策略以换取极致的吞吐量。在此级别下，查询操作既不施加任何读锁，也不依赖复杂的多版本并发控制（MVCC）历史链，一个事务可以直接穿透隔离边界，读取到其他并发事务在内存缓冲池（Buffer Pool）中已修改但尚未执行 COMMIT 持久化的数据页。这种“裸奔”式的读取机制导致其面临最严重的并发一致性灾难——脏读（Dirty Read）。本质上，RU 读取的是不具备事务原子性和持久性保证的“中间态脏数据”，一旦并发事务发生异常回滚，当前业务逻辑便会建立在凭空消失的虚假基础之上，因此在追求数据严谨性的现代工程实践中已基本绝迹。

（3）读已提交（RC）

RC 级别建立起了事务隔离的第一道实质性防线，它强制要求查询操作必须“越过”未提交的脏数据，严格读取其他并发事务已经成功 COMMIT 落盘后的版本，从而在根源上彻底斩断了脏读现象。在 InnoDB 引擎的底层实现中，RC 级别深度耦合了 MVCC 机制，其最核心的技术特征在于：事务中的每一次 SELECT 查询，都会动态生成一个全新的 Read View（读视图）快照。这意味着在同一个长事务的生命周期内，系统会不断刷新可见性边界以获取全局最新已提交的状态。正是这种“步步刷新”的底层逻辑，导致了**不可重复读（Non-Repeatable Read）**现象的发生——由于每次查询都在捕捉最新的提交记录，同一次事务前后的相同查询，极大概率会因为其他事务的横向修改或删除操作，而得到截然不同的结果集。

（4）可重复读（RR）

在 RR 级别下，如果您只是执行普通的 SELECT，MySQL 会使用 MVCC（多版本并发控制） 来给您提供一个历史快照，这样不管别人怎么插入，您看到的永远是快照里的数据，从而避免了幻读。

但是，如果您执行的是当前读（也就是加锁读，比如 SELECT ... FOR UPDATE，或者直接执行 UPDATE / DELETE 语句），MVCC 就失效了，因为这些操作必须对最新的数据进行操作。

为了防止其他事务在此期间“趁机”插入新数据引发幻读，InnoDB 祭出了终极武器——Next-Key Lock（临键锁）。

  • 技术本质：Next-Key Lock = 行锁 (Record Lock) + 间隙锁 (Gap Lock)。
  • 直白理解：行锁负责死死按住查询范围内已经存在的“实体数据行”；而间隙锁则像拉起了一道道警戒线，负责封锁这些数据行之间、以及首尾两端的“虚拟空隙”。

（5）串行化（Serializable）

定义： 事务串行执行，完全隔离。

问题： 性能极低（类似单线程排队）。

3.2 为什么互联网大厂偏爱 RC 级别

在超高并发的高级互联网应用中，往往会将默认的 RR 级别手动降级为 RC 级别，原因在于：

（1）减少锁冲突与死锁：RR 级别为了防止幻读，会大量使用 间隙锁 (Gap Lock)，锁住数据之间的空隙，极其容易引发并发死锁；而 RC 级别下没有间隙锁（除极少数特殊情况），锁的粒度更小，并发吞吐量更高。

（2）不可重复读在业务上可接受：对于大多数 C 端互联网业务，同一次会话中读到别人刚刚提交的最新数据，通常是符合业务直觉和逻辑的，不需要严格保证“可重复读”。

3.3 如何深刻理解 MVCC

MVCC 是 MySQL 实现无锁读（快照读），解决读写冲突的终极武器。它通过维护数据的多个历史版本，让读和写操作不冲突。

（1）底层支撑一：隐藏字段。InnoDB 会在每行数据后暗戳戳加上两个字段：trx_id（最后一次修改该行的事务ID）和 roll_pointer（回滚指针）。

（2）底层支撑二：Undo Log 版本链。每次修改数据前，MySQL 会把旧数据备份到 Undo Log 中，并通过 roll_pointer 把它们像链表一样串起来，形成历史版本链。

（3）底层支撑三：Read View (读视图)。这是一个“快照名单”。在事务执行查询的那一刻，系统会生成一份当前所有活跃（未提交）事务的 ID 列表。

  • 核心逻辑：当你读取一行数据时，拿着行上的 trx_id 去跟 Read View 规则比对。如果发现这个版本是别人还没提交的，就顺着版本链往下找，直到找到一个你“有资格”看到的历史版本。
  • RR 和 RC 的本质区别：RC 是每次执行查询操作前都生成一个全新的 Read View；而 RR 是在事务的第一次查询时生成一个固定的 Read View，一直用到事务结束。

四、 MySQL 日志系统与高可用架构

4.1 MySQL 的三大核心日志

（1）Redo Log (重做日志)：保证持久性 (Crash-Safe)。它是 InnoDB 引擎特有的。当修改数据时，先不急着刷磁盘，而是写到 Redo Log 中。就算数据库突然断电宕机，重启后只要看着 Redo Log 把没做完的操作重做一遍，数据就不会丢。

（2）Undo Log (回滚日志)：保证原子性与 MVCC。记录的是数据的相反逻辑操作（你 insert 它就记录 delete）。用于事务执行失败时的回滚，也是 MVCC 维持历史版本链的核心载体。

（3）BinLog (归档日志)：MySQL Server 层自带的日志，记录了所有修改数据库结构的 DDL 语句和修改数据的 DML 语句（不包括查询）。主要用于主从复制和数据恢复（如“删库跑路”后的按时间点恢复）。

4.2 主从复制机制

为了分摊数据库压力，通常一主多从，主库负责写，从库负责读。

（1）同步原理（三个核心线程）

  • 主库收到写请求修改数据后，写入自身的 Binlog。
  • 从库启动一个 I/O 线程，连接主库，请求读取 Binlog，并将其拷贝到从库本地的 Relay Log (中继日志) 中。
  • 从库启动一个 SQL 线程，不断读取 Relay Log 中的日志，并在本地重放执行，从而保持与主库的数据一致。

（2）延迟问题：由于网络原因或从库 SQL 执行慢，主从之间必然存在微小的延迟时间差。
