00 操作系统

jiwang4/16/26About 15 min计算机基础知识

一、 进程与并发控制机制

1.1 进程、线程与协程的深度剖析

（1）进程 (Process)：资源分配的基本单位

进程是程序执行时的一个实例。每个进程都有自己独立的内存空间（代码段、数据段、堆栈等）和系统资源。操作系统通过进程控制块（PCB）来管理进程。因为进程间相互独立，所以安全性高，但创建、销毁和切换进程时，需要保存和恢复各种寄存器和内存映射表，导致上下文切换开销非常大。

（2）线程 (Thread)：CPU调度的基本单位

线程是进程内的一个执行流，一个进程可以包含多个线程。同一进程内的线程共享进程的内存空间和资源（如全局变量、打开的文件），但每个线程有自己独立的运行栈和程序计数器（PC）。因为共享资源，线程间的通信更方便，且上下文切换时不需要切换内存地址空间，切换开销比进程小得多。但缺点是一个线程崩溃，可能导致整个进程崩溃。

（3）协程 (Coroutine)：用户态的轻量级线程

协程的调度完全由用户程序（即代码本身）自主控制，不涉及操作系统的内核态切换。它通过保存和恢复程序内部的状态来实现暂停和恢复执行。因为没有陷入内核态的开销，协程的切换速度极快，占用内存极小，非常适合处理极其庞大的高并发 I/O 密集型任务。

1.2 孤儿进程与僵尸进程

（1）孤儿进程及其处理

定义：父进程先于子进程结束，此时仍在运行的子进程就成了“孤儿进程”。

结果：操作系统有完善的兜底机制，孤儿进程会被 init 进程（PID 为 1，系统的老祖宗进程）接管。init 进程会负责等待它们完成并回收资源，因此孤儿进程对系统没有危害。

（2）僵尸进程及其危害

定义：子进程已经终止执行，但父进程还在运行，且父进程没有调用 wait() 或 waitpid() 去获取子进程的状态信息。此时子进程的进程描述符（PCB）仍然保存在系统中，这种状态的进程就是“僵尸进程”。

危害：僵尸进程不占用内存和 CPU，但会白白占用一个进程号 (PID)。系统的 PID 数量是有限的，如果产生大量僵尸进程，会导致系统无法分配新的 PID，从而无法创建新进程。

解决：找到并杀死产生僵尸进程的父进程，僵尸进程就会变成孤儿进程，从而被 init 收包回收。

1.3 进程调度算法核心策略

（1）批处理系统调度算法

  • 先来先服务 (FCFS)：非抢占式，按请求顺序执行。对长作业有利，对短作业不利（短作业可能需要等待很长时间）。
  • 短作业优先 (SJF)：非抢占式，优先调度运行时间最短的作业。能获得最小的平均等待时间，但如果长作业不断被短作业插队，会导致长作业“饥饿”。

（2）交互式系统调度算法

  • 时间片轮转 (RR)：抢占式。每个进程分配一个时间片，用完即被剥夺 CPU 放到队尾。公平且响应快，但时间片大小很难权衡（太大退化为 FCFS，太小则频繁切换导致开销过大）。
  • 多级反馈队列调度：目前公认较好的综合算法。设置多个优先级队列，优先级越高的队列时间片越小。新进程先进入最高优先级队列，如果一个时间片没执行完，就降级到下一个队列。既照顾了短作业的响应时间，又兼顾了长作业的执行。

二、 内存管理与死锁控制

2.1 内存的非连续分配机制

外部碎片 (External Fragmentation)： 发生在“动态分区分配”时。程序按需申请内存，运行完后释放。久而久之，内存条就像一块被咬得坑坑洼洼的饼干。特点是：空间在程序外部，太小了用不上。 （解决办法：内存紧凑/拼凑，把程序都往一头挪，把小碎块合并成大块）。

内部碎片 (Internal Fragmentation)： 发生在“固定分区”或“分页存储”时。操作系统为了管理方便，把内存切成固定大小的块（比如一页 4KB）。如果一个程序只需要 1KB，系统还是会强行分给它一个 4KB 的块。特点是：空间分给程序了，但在程序内部浪费掉了。

（1）分页存储管理

将物理内存分为大小相等的“物理块/页框”，将逻辑内存分为同样大小的“页”。通过页表将逻辑页映射到物理块。优点是不会产生外部碎片（内存利用率高），缺点是页表需要占用额外内存。

（2）分段存储管理

按程序的逻辑结构（如主函数段、子程序段、数据段）划分内存。这非常符合程序员的视角，方便代码的共享和保护。缺点是容易产生外部碎片。

（3）虚拟内存与局部性原理

虚拟内存使得程序认为它拥有连续可用的庞大内存，实际上数据被分割存放于物理内存和磁盘交换区。它的基石是局部性原理：

  • 时间局部性：被访问过的数据/指令，不久后很可能再次被访问（所以要有缓存机制）。
  • 空间局部性：被访问过的数据/指令附近的区域，不久后很可能被访问（所以可以整块读入内存）。

2.2 页面置换算法

（1）最佳置换算法 (OPT)

淘汰未来最长时间内不再被访问的页面。这是理论上的最优解，但因为无法预知未来，实际中无法实现，仅用作衡量其他算法的标杆。

（2）先进先出 (FIFO) 与 Belady 异常

淘汰最早进入内存的页面。实现简单，但性能差，且存在 Belady 异常：即分配的物理块数增加时，缺页率反而升高的反直觉现象。

（3）最近最久未使用 (LRU)

淘汰最近一段时间内最久没有被访问的页面。基于局部性原理，性能很好，是工业界最常用的算法。

2.3 死锁

（1）死锁产生的四个必要条件

必须同时满足以下四个条件才会发生死锁：

  1. 互斥条件：资源在一段时间内只能被一个进程独占。
  2. 请求与保持条件：进程已经保持了至少一个资源，但又提出了新的资源请求被阻塞，且对自己已有的资源死不放手。
  3. 不可剥夺条件：进程获得的资源在未使用完之前，不能被强行剥夺。
  4. 循环等待条件：发生死锁时，必然存在一个进程——资源的环形链。

（2）死锁的预防与避免

  • 死锁预防：在系统设计时就破坏上述四个条件之一。
  • 死锁避免（银行家算法）：在系统运行过程中，当进程请求资源时，先计算此次分配是否会导致系统进入“不安全状态”。如果安全才分配，否则让进程等待。

（3）死锁的检测与解除

允许系统发生死锁，但定期运行检测程序。一旦检测到死锁，通过资源剥夺（挂起某些进程抢夺资源）或撤销进程（强制杀掉死锁环路中的一个或多个进程）来打破僵局。

三、 系统底层机制与高并发 I/O 模型

3.1 运行模式与中断

（1）用户态 && 内核态

这是 CPU 的两种特权级别设计（Ring 0 为内核态，Ring 3 为用户态）。

  • 用户态：运行普通的应用程序，只能访问受限的内存空间，不能直接调用硬件资源。

  • 内核态：操作系统的核心运行状态，拥有最高权限，可以执行所有 CPU 指令，访问任何内存空间和底层硬件。

    这种设计是为了隔离保护，防止流氓软件或崩溃的程序带走整个操作系统。

（2）用户态切换到内核态的三种方式

  • 系统调用 (System Call)：程序主动要求操作系统提供服务（如读写文件、申请内存）。
  • 异常 (Exception)：程序运行出错（如除以零、缺页异常），触发 CPU 切换状态进行异常处理。
  • 外围设备中断：硬件设备（如网卡接收到数据、键盘被按下）向 CPU 发送中断信号，强行打断当前程序，切换到内核态执行中断处理程序。

（3）中断的处理流程

中断发生时，系统必须：保存当前现场（将寄存器、程序计数器等压入栈中） -> 根据中断向量表找到并执行对应的中断服务程序 -> 恢复现场并返回用户态继续执行。

3.2 I/O 多路复用：select、poll、epoll 演进史

I/O 多路复用使得单个线程就能同时监听多个网络连接（文件描述符 FD），是解决高并发网络问题的核心。

（1）select：早期的开荒者

  • 原理：将所有需要监听的 FD 放入一个集合（基于数组），每次调用 select，都要把整个集合从用户态拷贝到内核态。内核遍历这个集合，看有没有事件发生，如果有，再把集合拷贝回用户态，用户态再次遍历集合找到具体是哪个 FD 发生了事件。
  • 缺点：1. 连接数受限（默认最多只能监听 1024 个）；2. 频繁的用户态和内核态数据拷贝，开销大；3. O(N) 复杂度的轮询遍历，连接数越多效率越低。

（2）poll：突破数量限制

  • 原理：机制与 select 几乎一样，唯一的区别是底层采用链表代替了数组来存储 FD。
  • 优缺点：突破了 1024 个最大连接数的限制，但依然没有解决数据频繁拷贝和 O(N) 轮询遍历的性能瓶颈。

（3）epoll：高并发的终极方案 (Linux 专属)

epoll 彻底改变了设计思路，通过三个函数实现：

  • epoll_create：在内核态创建一个 epoll 对象（底层是一棵红黑树，用来存储要监听的 FD，以及一个就绪双向链表）。
  • epoll_ctl：用来向红黑树中增删改 FD，不需要每次都拷贝整个集合。更重要的是，它会为每个 FD 注册一个回调函数。
  • epoll_wait：当网卡收到数据时，触发回调函数，将发生事件的 FD 放入“就绪链表”。epoll_wait 只需要检查链表是否有数据并返回即可。
  • 优势：没有连接数限制；无需频繁整体拷贝数据；通过事件驱动而非轮询，获取活跃连接的时间复杂度是 O(1)。这也是 Nginx 和 Redis 单线程能支持百万并发的底层密码。

四、 Linux 常用命令

4.1 文件与目录操作管理

（1）基础文件操作

  • ls -lah：列出当前目录下所有文件（包含隐藏文件）的详细信息，且以人类可读的格式（MB, GB）显示大小。
  • cd / (切换到根目录), pwd (打印当前所在绝对路径)。
  • mkdir -p /a/b/c：递归创建多级目录。
  • rm -rf 文件/目录：强制递归删除，无确认提示（极度危险，敲下回车前务必三思）。

（2）文件内容查看与检索

  • cat / less：查看静态文件内容。less 支持上下翻页搜索，适合大文件。
  • tail -f xxx.log：排障必备，实时跟踪查看日志文件的最新尾部内容更新。
  • grep "关键字" xxx.log：在文件中搜索特定字符串，常配合管道符 | 使用（例：ps -ef | grep java 查找 java 相关进程）。

（3）权限控制与归属

  • chmod 755 文件名：修改文件权限（r=4, w=2, x=1。7代表属主可读写执行，5代表属组和其他用户可读可执行）。
  • chown user:group 文件名：更改文件的拥有者和所属群组。

4.2 系统状态与性能监控

（1）进程与 CPU 监控

  • top：运维第一神令。动态实时显示系统整体资源消耗，包括 CPU 负载、内存使用率、僵尸进程数量以及各个进程的资源占用排名。
  • ps -ef：静态快照式查看当前系统中运行的所有进程详细信息。
  • kill -9 PID：强行终止指定 PID 的进程。

（2）内存与磁盘监控

  • free -h：查看内存条的使用情况。关注 available 字段了解系统当前真实可用的内存。
  • df -h：查看各个磁盘分区的文件系统使用率，排查磁盘是否被打满。

（3）网络状态排查

  • netstat -ntlp：查看当前服务器有哪些端口正在被监听，以及占用该端口的程序 PID。
  • lsof -i:端口号：反向查询，查看某个具体的端口当前被哪个进程占用。
  • ping (测试网络连通性) 和 telnet ip 端口 (测试对方特定端口是否开放防火墙)。

五、 进程与线程的通信机制

5.1 进程间通信 (IPC, Inter-Process Communication)

因为每个进程都有自己独立的虚拟内存空间，进程 A 不能直接读写进程 B 的内存，所以必须借助操作系统内核来搭桥。常见的有以下七种方式：

（1）管道 (Pipe)

  • 原理：在内核中开辟一块缓冲区，数据以字节流的形式单向流动（半双工）。
  • 局限：只能用于具有亲缘关系的进程间通信（如父子进程、兄弟进程）。
  • 举例：Linux 命令中的 |（如 ps -ef | grep java），就是把前一个进程的输出作为后一个进程的输入。

（2）命名管道 (FIFO)

  • 原理：在文件系统中创建一个类型为管道的真实实体文件。
  • 突破：打破了亲缘关系的限制，任何毫无关系的进程只要能访问该路径，就能通过它进行通信。

（3）消息队列 (Message Queue)

  • 原理：存放在内核中的消息链表，进程可以向队列中添加消息，也可以从中读取消息。
  • 优势：克服了管道“数据无格式、只能按字节流读取”的缺点。数据被封装成一个个独立的消息块（有类型和边界）。
  • 缺点：通信不及时，且每次读写都需要经历用户态到内核态的拷贝过程。

（4）共享内存 (Shared Memory)

  • 原理：操作系统在物理内存中开辟一块空间，然后将这块空间同时映射到多个进程的虚拟地址空间中。
  • 优势：进程可以直接读写这块内存，完全不需要陷入内核态和数据拷贝，速度极快。
  • 注意：多个进程同时写同一块内存必然冲突，因此必须配合信号量来使用，以实现同步。

（5）信号量 (Semaphore)

  • 原理：本质上是一个在内核中维护的计数器，用来表示系统中可用资源的个数。
  • 定位：它不是用来传递复杂数据的，而是用来控制多个进程对共享资源（比如共享内存）的访问，实现互斥与同步。

（6）信号 (Signal)

  • 原理：唯一的异步通信机制。操作系统通过发送信号来通知进程发生了某种系统事件。
  • 举例：你在终端按下 Ctrl+C 产生了 SIGINT 信号；用 kill -9 PID 发送了强制终止的 SIGKILL 信号。

（7）套接字 (Socket)

  • 原理：不仅能用于同一台主机上的进程通信（Unix Domain Socket），更是跨网络、跨机器进程通信的唯一标准方式。

5.2 线程间通信与同步机制

因为同一个进程内的所有线程天然共享进程的代码段、数据段（全局变量）和堆（对象实例），所以它们互相传递数据非常容易，直接读写变量即可。它们面临的真正难题是同步（先后顺序）与互斥（排他访问）。

（1）互斥锁 (Mutex)

  • 原理：最基本的互斥手段。保证同一时刻只有一个线程能够获取锁并进入临界区（访问共享资源的代码段）。其他拿不到锁的线程只能阻塞等待。

（2）读写锁 (Read-Write Lock)

  • 原理：对互斥锁的性能优化。分为“读锁”和“写锁”。
  • 规则：读读不互斥，读写互斥，写写互斥。非常适合“读多写少”的高并发场景。

（3）信号量 (Semaphore)

  • 原理：与进程的信号量类似，但作用于线程。它可以控制同一时刻允许多个线程访问同一个共享资源。
  • 举例：停车场有 5 个车位（信号量初始值为 5），来一辆车减一，变为 0 时后续的车辆（线程）只能在外面排队等待。

（4）条件变量 (Condition Variable)

  • 原理：当线程需要的某个特定条件没有满足时，它会主动交出互斥锁并进入阻塞状态，等待其他线程改变条件并“唤醒”它。
  • 注意：条件变量必须和互斥锁配合使用，以防止多线程环境下的竞态条件。
  • 举例：Java 中的 wait() 和 notify() 机制，或者是生产者-消费者模型中队列满/空时的等待与唤醒。
