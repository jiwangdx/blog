01 计算机网络

jiwang4/16/26About 9 min计算机基础知识

一、 Http & Https

1.1 HTTP 协议核心基础

应用层协议，基于 TCP 传输。其最大的特点是无状态，为了在多次请求间“记住”用户，引入了 Cookie 和 Session 机制，或是现代化的 JWT (JSON Web Token) 方案。

HTTP 报文结构：由请求行（方法/URL/版本）、请求头（Headers）和请求体（Body）组成。

1.2 HTTP/1.0、HTTP/1.1、HTTP/2.0、HTTP/3.0 的区别

HTTP/1.0：每次请求都建一条新通道

每次向服务器请求资源（哪怕是一张小图片）都必须经历耗时的 TCP 三次握手来新建通道，传输完成后立刻断开，这种频繁建连和断连的操作导致网络开销极大，效率极其低下。

HTTP/1.1：连接可以复用，减少重复开销

引入了 Keep-Alive 长连接机制，允许同一条 TCP 通道连续发送多个请求而无需反复重连；但服务器必须严格按照请求的先后顺序返回结果，一旦前面的请求卡住，后面的响应全得排队死等，这就是“应用层队头阻塞”。

HTTP/2.0：一条通道可以同时跑多个请求

通过“二进制分帧”将数据切碎，让多个请求和响应在同一条 TCP 连接上交错混发，解决了应用层的排队问题；但由于底层仍然受制于要求绝对顺序的 TCP 协议，一旦网络丢包，整个通道就会停下等待重传，暴露出“TCP 队头阻塞”的致命软肋。

HTTP/3.0：换通道协议，彻底解决堵塞

彻底抛弃了历史包袱沉重的 TCP，改用基于 UDP 开发的 QUIC 协议，不仅将连接建立与加密握手合二为一（极速秒连），还让所有并发的请求流相互独立，哪怕某个包在半路丢失，也绝不影响其他请求的正常传输，实现了完美的“真并发”。

1.3 从「敲下一个 URL」到「页面出现在屏幕」整条链路全景

  • 浏览器通过DNS解析服务器IP地址，三次握手建立连接，如果是HTTPS，则会进行加密，证书认证。

  • 浏览器构建请求，通过网络发送到服务器 。

  • 服务器获取到请求，向上传递到应用层解析请求，获取处理方法和资源 。

  • 服务器构建响应报文，把获取的资源界面通过网络再发送给浏览器 。

  • 浏览器收到响应，解析资源显示到界面上。资源获取完毕，通过四次握手断开连接。

1.4 HTTP vs. HTTPS 有什么区别?

  • HTTP是明文传输，HTTPS加了TLS协议，是加密传输，更安全；

  • HTTP只需要TCP三次握手过程，而HTTPS还增加了TLS握手过程；

  • HTTP端口号是80，HTTPS端口号是443；

  • HTTPS需要通过CA（证书权威机构）申请数字证书，确保服务器是可信的。

1.5 HTTPS 的 RSA 实现

HTTPS 握手的全过程，本质上是为了解决两个核心问题：第一是‘验明正身’，也就是证书校验，防止中间人攻击；第二是‘协商暗号’，也就是密钥交换，确保后续通信不被窃听。整个流程可以分为三个阶段来理解。

打招呼阶段

  • 客户端先给服务器发一个客户端随机数和支持的加密套件。
  • 服务器收到后，回复一个服务器随机数、选定的加密算法，并且把自己的数字证书发过来。

证书校验阶段

  • 客户端拿到证书后，会用系统内置的 CA 机构公钥去验证这个证书的数字签名。这一步非常关键，它确认了服务器的真实身份，确保我们没有连到钓鱼网站。

密钥协商阶段

  • 证书校验通过后，客户端会生成第三个随机数，叫做预主密钥。
  • 客户端用证书里的服务器公钥把这个预主密钥加密，发给服务器。服务器用自己的私钥解密，这样双方就都拿到了这个预主密钥。
  • 现在，双方手里都有了客户端随机数、服务器随机数和预主密钥。双方各自用这三个变量，算出最终的对称加密会话密钥。之后的通信，就全部用这个会话密钥进行高效的对称加密。

既然公钥和私钥是成对生成的，那黑客拿到了公钥，为什么不能通过公钥反向推导（破解）出私钥呢？大数分解难题，私钥的两个小数都是质数，也就是说答案是唯一的。

二、 TCP 三次握手与四次挥手

2.1 三次握手：连接的建立

（1）服务端处于监听状态，客户端向服务端发送第一次连接请求，会随机生成一个客户端序列号，SYN报文置1，表示发起连接（第一次握手）；

（2）服务端收到SYN报文后，也随机生成一个服务端的序列号，将收到的客户端序列号+1，ACK置1，将报文全部发给客户端（第二次握手）；

（3）客户端收到服务端的报文后，将服务端的序列号+1，然后再次发送ACK报文给客户端，客户端处于已建立状态，这次握手可以携带数据到服务端（第三次握手）；

（4）服务端收到应答报文后，也处于已建立状态

2.2 四次挥手：连接的释放

数据传输完毕后，双方都可释放连接。最开始的时候，客户端和服务器都是处于ESTABLISHED状态，然后客户端主动关闭，服务器被动关闭。

第一次挥手 客户端发出连接释放报文，并且停止发送数据。释放数据报文首部，FIN=1，其序列号为seq=u（等于前面已经传送过来的数据的最后一个字节的序号加1），此时，客户端进入FIN-WAIT-1（终止等待1）状态

第二次挥手 服务器端接收到连接释放报文后，发出确认报文，ACK=1，ack=u+1，并且带上自己的序列号seq=v，此时，服务端就进入了CLOSE-WAIT 关闭等待状态

第三次挥手 客户端接收到服务器端的确认请求后，客户端就会进入FIN-WAIT-2（终止等待2）状态，等待服务器发送连接释放报文，服务器将最后的数据发送完毕后，就向客户端发送连接释放报文，服务器就进入了LAST-ACK（最后确认）状态，等待客户端的确认。

第四次挥手 客户端收到服务器的连接释放报文后，必须发出确认，ACK=1，ack=w+1，而自己的序列号是seq=u+1，此时，客户端就进入了TIME-WAIT（时间等待）状态，但此时TCP连接还未终止，必须要经过2MSL后（最长报文寿命），当客户端撤销相应的TCB后，客户端才会进入CLOSED关闭状态，服务器端接收到确认报文后，会立即进入CLOSED关闭状态，到这里TCP连接就断开了，四次挥手完成。

为什么客户端要等待2MSL？保证客户端发送那个的第一个ACK报文能到到服务器，因为这个ACK报文可能丢失，并且2MSL是任何报文在网络上存在的最长时间，超过这个时间报文将被丢弃，这样新的连接中不会出现旧连接的请求报文。

三、 TCP 可靠性保证

确认应答和序列号：接收方在收到数据后必须返回确认，并携带下一个期望收到的序列号

超时重传机制：发送方在一定时间内没有收到ack，就会重新发送数据。

连接管理：通过三次握手建立连接，四次挥手释放连接，保证了双向连接的稳定性。

流量控制：通过滑动窗口根据接收方的处理能力来调整发送速率，避免了接收方的缓冲区溢出。

拥塞控制：通过慢启动，拥塞避免算法和快速重传等机制，根据网络状况动态调整发送速率。

四、 TCP 粘包和拆包

TCP 底层没有消息边界的概念。需要在消息头部加上一个长度字段。接收端每次先读取长度，然后根据这个长度去截取后面完整的数据块，这样就能精准地把消息拆分开来。

五、 网络安全与性能

5.1 SYN 攻击与防御

正常的 TCP 连接要三次握手。SYN 攻击就是，坏人派了成千上万个小弟，跑过来只打个招呼（发 SYN），然后就原地消失不回话了。服务器为了等这些小弟，把“接待室（半连接队列）”全占满了，导致真正的客户进不来。

SYN 攻击是一种典型的资源耗尽型攻击，打中了 TCP 握手需要维护连接状态的弱点。作为后台开发，我们一般不直接在代码里防这个，而是依靠服务器操作系统层面的配置（比如开启 SYN Cookies，生成个验证码然后打回去等第二次互动），或者直接在我们的 Java 服务前面挡一层Web 应用防火墙来过滤这种恶意流量。

**SYN Cookies 防御的绝妙之处在于抓住了黑客“不敢暴露真身”的死穴：**由于黑客为了隐藏自己必须伪造源 IP，服务器算出的验证码（Cookie）最终发给了毫无关联的假 IP（路人），导致黑客根本收不到验证码，自然无法完成三次握手（服务器零内存损耗）；而一旦黑客敢用真实 IP 强行收发验证码完成握手，他的真实身份不仅会瞬间暴露并被防火墙彻底拉黑，他自身机器的资源也会因为建立真实连接而被反向耗尽。

5.2 CDN (内容分发网络)

CDN（内容分发网络）本质上不是几台散落的“小服务器”，而是一套依托 DNS 智能调度（GSLB）的分布式边缘缓存系统。 当用户请求资源时，底层的 DNS 会将其精准重定向到物理距离最近的边缘节点；如果节点内已有资源则直接极速返回（缓存命中），若没有则由节点代替用户向核心机房拉取数据并存下（回源）。这套机制不仅让用户享受到了毫秒级的加载速度，更像一面巨大的流量护盾，替脆弱的后端 Java 源站挡下了绝大部分的高并发压力与恶意攻击。
