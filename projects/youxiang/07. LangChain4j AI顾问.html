07. LangChain4j AI顾问

jiwang2/15/26About 19 min项目《优享生活圈》ragLangChain4jMCP

一、需求实现

  • 设计并实现分层多轮会话管理机制，基于Redis构建短期记忆，MySQL实现中期记忆和长期记忆，支持会话列表管理。

  • 构建架构知识库，实现商家信息的自动向量化处理和定期更新，提升语义检索准确性。

  • 开发工具集，实现商家信息查询、预约下单等业务功能与大模型的无缝集成。

  • 优化系统性能，实现增量更新机制，定期全量更新兜底，结合分页查询，减少计算资源和内存的消耗，提升响应速度。

背景

现在的大模型是非常热门的一个方向，业界也有很多的一些传统项目都在结合AI进行这种智能化升级。所以我就思考自己这个项目是不是也能结合AI去做一些帮助用户的功能。

我的想法是通过Java调用大模型给用户提供一个智能AI顾问，它有点类似于一个GPT对话式的顾问，

那它所能提供的功能，例如：

1. 查询商家的信息
2. 查询商家的优惠券信息
3. 查询用户拥有的优惠券信息
4. 预约到店消费服务
5. 查询用户到店消费服务的预约详情
6. 结合具体商家的信息，给用户做店铺推荐或玩乐安排等

为了实现智能AI顾问，首先我就去学习了这个Java调用大模型的工具框架，其中印象比较深刻的是

  • 像用户的一些问题，要结合平台的文档或数据才能够回答，这时候就需要使用，
    • 我需要把平台的文档或数据向量化存储到RAG知识库(向量数据库)
    • 当用户发出问题后，Java服务会
    • 得到知识片段后，Java服务需要结合用户的问题以及知识库中检索到的知识片段组织要发送给大模型的消息
    • 大模型接收到消息后会同时根据用户的问题、知识库检索到的知识片段以及自身的知识储备，生成对应的结果响应给Java服务，最终再返回给用户

可以利用rag来做个店铺推荐的服务。

在智能顾问的 RAG 设计中，最初采用 PDF 静态知识库方式存储商家信息，这在商家数量少、信息变化不频繁时可以正常工作，也实现了基本的店铺问答和推荐功能。

但是随着业务发展，考虑到商家信息实际上是动态变化的——有新的商家入驻，已有商家的优惠、营业时间或其他信息也会更新。使用 PDF 做静态知识库就存在两个问题：

  • 更新不及时，新加入或修改的商家信息不能立即被模型检索到；
  • 全量向量化成本高，每次全量处理所有商家信息会消耗大量 Token，浪费计算资源。

为此，将方案优化为。首次运行时，进行全量向量化，将所有商家信息生成 embedding 并写入向量库；后续定时任务只读取 update_time 在时间窗口内的新增或更新商家，实现****，只对变动数据进行 embedding。

这种方式的好处是：

  • 降低 Token 消耗和计算成本，不必每次都全量处理；
  • 保证向量库及时更新，新入驻或变更的商家信息能被快速检索；
  • 具备可扩展性，适应商家数量增长和业务扩展。

后来在看代码时，我才发觉自己从数据库读取商家数据时使用的是一次性 List，由于商家表比较大，这会占用大量内存，容易造成性能问题。 于是我改成使用，每次只读取一部分数据，这样既降低了内存压力，又保证了数据处理的稳定性。

  • 可以智能地根据用户的输入调用我们准备好的工具方法。
    • 首先我需要准备一些工具方法，例如查询商家信息、新增预约信息等等的方法
    • 当用户发出问题后，Java服务需要组织提交给大模型的数据，而这些数据中需要描述清楚Java服务中有哪些函数能够被大模型调用。每一个函数的描述都包含三个部分，方法名称、方法作用、方法入参。
    • 当Java服务把这些数据发送给大模型后，大模型会先根据用户的问题以及上下文拆解任务，从而判断是否需要调用函数。如果有函数需要调用，则把需要调用的函数的名称，以及调用时需要使用的参数准备好一并响应给Java服务。
    • Java服务接收到响应后需要执行对应的函数，得到对应的结果，接下来把得到的结果和之前信息一块组织好再发送给大模型。最后再响应给用户。

注意

使用Redis支持会话记忆是什么意思？

大模型不具备记忆能力，每次会话都是独立的。要想让大模型产生记忆的效果，唯一的方法就是把之前聊天的内容和新的内容一起发送给大模型。也就是说我们需要存储一个会话的互相发过的消息。

这个会话的消息是需要存储的，且不能只存储在内存里，否则如果后端重启就消失了，所以需要持久化，所以存储在Redis中（String存储即可），有效期设置一天足够。

二、核心概念理解

2.1. MCP（Model Context Protocol）

MCP是一种用于规范大模型与应用系统间上下文交互的协议标准，核心解决多轮对话中的上下文管理、工具调用和状态同步问题。虽然我这个项目未直接使用MCP，但实现了类似的核心功能：

首先，在会话上下文管理方面，项目通过Redis自定义 RedisChatMemoryStore ，将对话历史序列化存储，确保多轮对话的连贯性，这与MCP的一致，只是实现方式更偏向业务定制。

其次，在工具调用能力上，项目通过LangChain4J的 @Tool 注解定义商家信息查询、预约下单等工具，实现模型与业务系统的交互，这对应MCP的，但我们直接基于框架注解实现，开发更快捷。

最后，从技术趋势看，MCP的标准化优势在于降低多模型集成的复杂度，适合大型系统；而我们的实现方式更灵活，适合中小规模项目的快速迭代。未来若需扩展多模型支持，引入MCP相关框架会是不错的选择。

2.2. RAG（Retrieval-Augmented Generation）

RAG是一种技术架构，通过检索知识库信息来增强模型回答的准确性。和MCP相比，但可以协同工作：

比如在项目中，实现了基于向量存储的RAG架构，用于检索商家信息。如果引入MCP，可以：

1. 标准化检索结果的格式，确保模型能更准确理解知识
2. 规范工具调用接口，简化商家信息查询、预约下单等功能的集成
3. 统一上下文传递机制，提升多轮对话的连贯性

3.如果大模型要记很久的消息呢，我通过了解的话，大概是通过分级存储的方式去实现。

所以我尝试实现了一下，建了一张专门存储会话信息的表，定时任务轮询去检查有没有过期信息，有则删除

3.4 RAG架构的动态知识库管理

// 向量存储服务
@Service
public class EmbeddingStoreService {
    @Autowired
    private ShopMapper shopMapper;
    @Autowired
    private EmbeddingModel embeddingModel;
    @Autowired
    private EmbeddingStore store;

    // 应用启动时全量加载
    @PostConstruct
    public void init() {
        fullUpdate();  // 首次全量加载
    }

    // 定期增量更新
    @Scheduled(fixedRate = 10000)
    public void periodicIncrementalUpdate() {
        incrementalUpdate();  // 增量更新
    }

    // 全量更新方法
    private void fullUpdate() {
        // 1. 查询所有商家数据
        List<Shop> shops = shopMapper.selectList(new QueryWrapper<>());
        // 2. 转换为文档
        List<Document> documents = convertToDocuments(shops);
        // 3. 向量化并存储
        ingestDocuments(documents);
    }

    // 增量更新方法
    private void incrementalUpdate() {
        // 1. 查询时间窗口内的更新数据
        List<Shop> shops = shopMapper.selectList(queryWrapper);
        // 2. 转换为文档
        List<Document> documents = convertToDocuments(shops);
        // 3. 向量化并存储
        ingestDocuments(documents);
    }
}

五、rag优化

5.1. 增量更新的策略

巧妙的是，无论是新增还是基于旧数据的修改，都能解决，因为代码逻辑是查询向量库是否存在与对应商户ID相关的向量，具体方法为inMemoryStore.remove(shopIdStr)。如果 ID 不存在， 该方法不会报错 ，只会什么都不做

但是！！！删除不行，为了解决删除的问题，在向量管理中，我设计了一个 映射表 来记录商铺ID和向量ID的对应关系。，先对比映射表和数据库，找出已被删除的商铺，然后清理对应的向量和映射表记录。

每次增量更新时：

  1. 查询映射表中所有商铺ID
  2. 查询数据库当前所有商铺ID
  3. 对比两边，找出在映射表中存在但数据库已不存在的 → 孤立记录
  4. 删除这些孤立记录的向量 + 映射表记录
// 每次增量更新时，检测并删除被删除的商铺向量

function detectAndRemoveOrphanedEmbeddings():
// 1. 获取映射表中所有商铺ID
mappingShopIds = SELECT shop_id FROM shop_embedding_mapping

// 2. 获取数据库当前所有商铺ID
dbShopIds = SELECT id FROM shop

// 3. 找出孤立的商铺（在映射表中存在，但数据库已删除）
orphanedIds = mappingShopIds - dbShopIds

// 4. 删除孤立向量和映射表记录
for shopId in orphanedIds:
embeddingId = SELECT embedding_id WHERE shop_id = shopId
vectorStore.remove(embeddingId)      // 删除向量
DELETE FROM mapping WHERE shop_id = shopId  // 删除映射记录

注意

在前期实现增量更新时，我们采用了的方式进行数据扫描。目前的处理逻辑同样嵌套在分表查询流程之下：先按分表策略查询 shop 表数据，再基于分表结果执行差集计算。这种方式能够有效适配大数据量场景，降低单次数据加载规模，使整体内存占用更加可控，提升系统的稳定性与可扩展性。

5.2. 对rag的深度优化（0315更新）

关于QA切分的坑：
"当时处理一个FAQ文档，直接按段落切分后发现，用户问'怎么退款'，结果召回的是某个问题的前半段，答案没召回到。后来改成按QA对绑定——把问题和答案作为一个整体chunk，同时单独把问题字段也存一份用于精确匹配，召回率一下就上来了。"

关于边界丢失的坑：
"递归切分时发现，有些句子的后半句被切到下一个chunk了，导致语义不完整。后来加了15%的overlap，让相邻chunk有重叠，这个问题基本解决。同时我们也在chunk里保留了上下文的元数据，比如章节标题，这样即使用到的是中间的chunk，LLM也知道它来自哪一章。"

（1）数据清洗：RAG的“第一道关口”

在实际场景中，数据清洗是必须的环节，通常放在切分之前。可以按“文档级清洗”和“段落级清洗”两个层面来考虑：

  • 文档级清洗（PDF/Word/网页）：

    • 去除噪声：比如页眉页脚、页码、目录、超链接、广告注释，以及从PDF复制时产生的多余换行和空格。

    • 格式统一：将不同来源的文档统一转为TXT或Markdown，方便后续处理。

    • OCR（光学字符识别）预处理：如果是扫描件，需要先做OCR识别，并修复识别错误的文字。

  • 段落级清洗：

    • 特殊字符过滤：去掉乱码或对检索无意义的符号。

    • 语义去重：如果数据是爬取的，可能存在大量重复内容，需要通过算法（如MinHash）去重，避免检索结果千篇一律。

    • 隐私信息脱敏：在进入知识库前，用占位符替换手机号、身份证号等敏感信息。

（2）必须考虑的“进阶”问题

除了数据清洗，还有几个关键点是在实际场景中必须考虑的，它们直接决定了RAG系统的上限。

A. 检索效果优化

单纯依靠top-k返回向量片段，有时效果不理想。可以引入更精细的策略：

  • 混合检索（Hybrid Search）：只用向量检索可能漏掉关键词。**“向量检索” + “关键词检索（如BM25）”**结合，能同时兼顾语义和字面匹配。
  • 重排序（Re-ranking）：第一次检索（top-k可能设为50）返回一个较大的候选集，然后用一个更精准的模型（如bge-reranker）对结果重新打分排序，最后返回top-3给大模型。这能显著提升答案质量。
  • 元数据过滤：在检索时，先根据元数据过滤。比如只搜索“2024年”的文档，或只搜索“某商家”的数据，能大幅提高检索的准确性。
B. 文档结构与元数据
  • 元数据附着：切分后，每个chunk需要携带来源信息（如文件名、章节标题、页码、作者、时间戳）。这样，当大模型使用某个chunk时，可以通过上下文知道它出自哪里。比如在回答末尾自动生成引用链接，就是靠元数据实现的。
  • 结构化数据索引：对于商家库的“按记录切分”，不能直接把数据库行简单丢进去。通常需要做字段拼接，把行数据转成一段通顺的描述文本，比如：“商家A成立于2020年，主营电子产品，位于北京……”，这样向量化效果会更好。
C. 上下文“窗口”管理

chunk_size参数，需要和大模型的能力联动考虑：

  • 块的大小与LLM窗口的配合：如果LLM的上下文窗口很小，但你的chunk有2000字，一次给进去5个，很可能超长导致截断。需要根据最终选用的模型来反推chunk的设计。
  • 多路召回：如果一个问题需要综合“商家信息”和“通用知识库”的信息，通常需要分别从两个库检索，然后将结果合并后再给LLM，而不是只从一个库找。
D. 评估与维护

RAG上线不是终点，持续的维护必不可少。

  • 如何评估效果好：建立一套测试集（QA对）。每次修改清洗逻辑、切分策略或检索参数后，都需要跑一遍测试集，看看答案准确率是上升还是下降。
  • 知识更新：数据会过期。需要设计更新机制。比如商家库变化后，是每天定时重建索引，还是实时更新？PDF文档新增版本后，旧版本如何处理？

（3）一个完整的RAG流程

  • 数据加载：读取PDF、DB等源数据。
  • 数据清洗：去噪、去重、格式化、脱敏。
  • 文本切分：根据场景（递归、QA、按记录）进行切分，附加元数据。
  • 向量化：将切分好的文本块通过Embedding模型转为向量。
  • 索引构建：存入向量数据库，并建立元数据索引。
  • 用户查询：收到问题。
  • 查询改写/优化：对问题进行改写。
  • 检索：执行混合检索 + 元数据过滤。
  • 重排序：对检索结果进行精排。
  • 生成：将问题和精选出的chunk + 历史对话一起交给LLM，生成最终答案。

6.4 LLM 判断调用工具的原理

LLM 判断是否需要调用工具的过程，实质上是基于训练时学到的模式预测过程：

第一步：Prompt 构建

LangChain4j 会自动将工具描述转换为 Prompt 中的工具说明：

你可以使用以下工具：
- shopTool: 根据商家名称查询商家信息
- reservationTool: 预约到店消费服务
- voucherTool: 根据商家名称查询商家的优惠券信息

这些描述来源于：

  • @Tool("...") 的值 → 工具功能描述
  • @P("...") 的值 → 参数含义说明

第二步：LLM 输出预测

LLM 接收到完整的 Prompt 后，会预测最可能的输出形式。当用户说"帮我查万科商场的优惠券"时，LLM 会：

  1. 匹配用户意图与工具描述
  2. 预测应该调用 findVoucherByShopName 函数
  3. 提取参数 shopName = "万科商场"

第三步：返回结构化调用请求

LLM 返回的不是自然语言，而是 JSON 格式的函数调用请求：

{
  "name": "findVoucherByShopName",
  "arguments": {
    "shopName": "万科商场"
  }
}

6.6 关键设计要点

设计要点说明
工具描述@Tool 注解的描述要清晰说明工具用途
参数描述@P 注解要准确说明参数含义和格式要求
功能独立各工具功能应独立，避免重叠导致 LLM 困惑
返回类型有返回值的工具会被 LLM 用于生成回答

本系统的工具调用机制完全满足单应用内部的 AI 咨询服务需求，无需引入 MCP 协议。若后续需对接外部独立服务，可考虑扩展 MCP 支持。
