### 二、 ArrayList 核心机制与线程安全

### 2.1 ArrayList 的基础设计

#### **（1）建议指定初始化容量的原因**

ArrayList 底层由动态数组实现。如果能在 `new ArrayList<>(capacity)` 时预估并设置合适的初始容量，可以**避免数组在添加元素时触发频繁的扩容机制**。每次扩容都需要在内存中创建新数组并进行数据拷贝（`Arrays.copyOf`），这会消耗性能并可能造成内存碎片。避免无效扩容不仅能提升运行效率，也能节省内存空间。

#### **（2）扩容因子默认为 1.5 倍的原因**

这是在“内存使用率”和“性能损耗”之间权衡的经验值（源码中为 `oldCapacity + (oldCapacity >> 1)`），适合大多数场景：

- **如果扩容因子太大**（如 2 倍）：可能申请了大量内存但使用不到，造成严重的内存浪费。
- **如果扩容因子太小**（如 1.1 倍）：会导致频繁触发扩容操作，带来高昂的性能开销。

### 2.2 ArrayList 的线程安全问题

#### **（1）为什么是不安全的？**

如果 ArrayList 被用作多线程共享数据（如单例对象的成员变量、静态变量），并发修改时会产生竞态条件：

- **元素覆盖**：两个线程同时执行 `add()`，可能由于拿到了相同的数组下标而导致后一个元素覆盖前一个。
- **数据丢失或越界**：多线程同时触发扩容机制时，可能导致底层数组的状态混乱，抛出 `ConcurrentModificationException` 或数组越界异常。

#### **（2）解决方案**

- 使用 `synchronized` 代码块将操作 ArrayList 的逻辑包裹起来。
- CAS乐观锁
- **推荐**：使用并发包下的 `CopyOnWriteArrayList`。

### 2.3 CopyOnWriteArrayList（写时复制）

#### **（1）实现原理**

底层依然基于数组实现。采用了**读写分离**的设计思想：

- **读操作**：直接在原数组上进行，完全不加锁。
- **写操作**：写操作（如 `add`/`set`）会先**加锁**（通常是 `ReentrantLock`），然后将原数组复制出一个新数组，在新数组上完成修改，最后将底层数组的引用指向这个新数组。

#### **（2）优缺点分析**

- **优点**：允许并发读而不阻塞，极大提高了读性能，非常适合**读多写少**的并发场景。
- **缺点**：每次写操作都要复制数组，极其消耗内存（可能触发频繁的 GC）；并且读到的可能是旧数据，**不适合实时性要求极高**的场景。

### 三、 HashMap 核心机制与线程安全

### 3.1 HashMap 的底层原理

#### **（1）底层数据结构演进**

- **JDK 1.7**：数组 + 链表。
- **JDK 1.8**：数组 + 链表 + 红黑树。当链表长度 >=8且 数组长度 >=64时，链表会转化为红黑树，将查询时间复杂度从O(N)优化到 O(log N)。

<img src="https://jiwang.cc.cd/PicGo/java/javase/2026-1776339289-f14753.jpg" alt="img" style="zoom:80%;" />

#### **（2）添加元素（put）流程**

1. 通过公式 `(数组长度 - 1) & 键的哈希值` 计算出元素应存入的数组索引（桶位置）。
2. 如果该位置为 `null`，直接创建节点存入。
3. 如果该位置不为 `null`（发生哈希冲突），则遍历该桶对应的链表或红黑树。
4. 调用 `equals()` 方法比较键。如果键一致，则覆盖旧的 `value` 并返回。
5. 如果没有相同的键，则将新节点加入结构中（JDK 1.7 使用**头插法**，JDK 1.8 改用**尾插法**防止成环）。

#### **（3）扩容加载因子（为什么是 0.75？）**

当 HashMap 中的元素数量达到 `数组长度*0.75` 时就会触发扩容。

- 这是一个基于**泊松分布**统计规律的折中值。在这个负载因子下，哈希冲突的概率显著降低（链表长度达到 8 的概率极小）。它能在“减少哈希冲突（保持高查询效率）”和“控制内存占用”之间达到最佳平衡。

#### **（4）扩容机制（为什么必须是 2 的次幂？）**

计算存储下标的公式是 `(n - 1) & hash`。

- 当数组长度 `n` 是 2 的次幂时，`n - 1` 的二进制表现形式全是 1（例如 16-1 就是 1111）。
- 此时进行按位与（`&`）运算，结果完全取决于 `hash` 值的低位特征，这相当于高效地实现了取模运算（`hash % n`），并且能最大程度地保证元素在数组中分布均匀，降低哈希碰撞概率。

### 3.2 HashMap 的线程安全问题

#### **（1）多线程下的隐患**

- **JDK 1.7**：并发扩容时，由于头插法的特性，极易造成链表死循环（环形链表）或数据丢失。
- **JDK 1.8**：虽然解决了死循环问题（改用尾插法），但在并发插入元素到空桶时，依然会发生**数据覆盖**的情况，且无法保证 `size` 统计的准确性。

#### **（2）解决方案**

- 使用 `Hashtable` 或 `Collections.synchronizedMap()`（全局加锁，性能差，不推荐）。
- **推荐**：使用 `ConcurrentHashMap`。

### 3.3 ConcurrentHashMap（并发哈希表）

#### **（1）JDK 1.7 时代：分段锁设计**

底层采用 `Segment` 数组 + `HashEntry` 数组 + 链表。默认将整个哈希表分为 16 段（`Segment`，继承自 `ReentrantLock`），每段有一把独立的锁。最多支持 16 个线程并发操作不同的段，显著降低了锁竞争。

![img](https://jiwang.cc.cd/PicGo/java/javase/2026-1776339687-6e121a.png)

#### **（2）JDK 1.8 时代：CAS + synchronized**

废弃了分段锁，底层结构与 1.8 的 HashMap 保持一致（数组 + 链表 + 红黑树）。锁的粒度进一步细化到了**单个桶的头节点**。

- 添加元素时，若数组未初始化，用 **CAS** 安全地初始化数组。
- 若目标桶位置为空，用 **CAS** 以无锁的方式将新节点放入桶中。
- 若目标桶位置已有元素（发生冲突），则使用 **`synchronized`** 锁住该桶的头节点，然后再进行链表遍历/红黑树插入操作。

<img src="https://jiwang.cc.cd/PicGo/java/javase/2026-1776339289-dba029.png" alt="img" style="zoom: 50%;" />

#### **（3）为什么混合使用 CAS 与 synchronized**

- **CAS（无锁乐观机制）**：适合执行**时间极短、竞争不激烈**的任务（如初始化数组、给空桶赋值）。如果是耗时较长的任务，CAS 失败会导致线程不断自旋尝试，白白浪费 CPU 资源。
- **`synchronized`（悲观锁）**：适合执行**耗时较长、竞争可能激烈**的任务（如遍历链表比较 key、链表转红黑树等）。加锁后，拿不到锁的线程会进入阻塞休眠状态，将 CPU 资源让给其他线程，避免了无意义的自旋消耗。两者的结合做到了性能的极致压榨。

### 3.4 为什么要重写 `equals` 和 `hashCode`？

重写的核心目的是将对象的唯一性判定从**物理坐标（它是谁，基于内存地址）转变为逻辑内容（它是什么，基于属性值）**；通过让 `hashCode` 负责“初筛定位”以确保相同内容落入同桶，让 `equals` 负责“精准识别”以确保内容一致即视为重复，从而打破“不同内存地址即不同对象”的默认规则，实现业务层面的去重逻辑 。
