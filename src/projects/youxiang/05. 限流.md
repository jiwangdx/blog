## **一、滑动窗口限流**
### 1.1 整体架构

采用注解、AOP 和 Redis 相结合的方式，实现无侵入式的限流组件。该组件支持三种限流维度：**全局限流用于秒杀接口防止系统过载，用户限流用于防止恶意刷券，IP限流用于抵御爬虫。**通过这种方式，既能保证系统的可用性和稳定性，又能灵活应对不同场景的限流需求。

### 1.2 限流算法的选择

在设计限流组件时，需要选择合适的限流算法。常见的算法有固定窗口、滑动窗口、漏桶和令牌桶四种。

固定窗口算法存在临界问题，在窗口切换的瞬间允许两倍流量通过，不够精确。漏桶算法以固定速率处理请求，虽然能够平滑流量，但无法应对突发流量。即使系统有空闲资源，请求也只能排队等待，在秒杀场景下会导致大量请求被丢弃，用户体验较差。令牌桶算法允许一定程度的突发流量，但存在脉冲攻击的风险——如果桶中积累了令牌，恶意用户可以在瞬间发送大量请求，挤占正常用户的资源。

综合考虑后，最终选择滑动窗口算法。它将时间窗口细分为多个小窗口，随着时间推移，旧窗口数据自动过期，统计的是滑动窗口内的总请求数。滑动窗口既能提供精确的控制，又能有效抑制突发流量，即使攻击者试图在短时间内发起大量请求，也会因为每个窗口都有上限而被限制住。

### 1.3 核心实现：基于 ZSet 的滑动窗口

滑动窗口限流的核心是使用 Redis 的 ZSet（有序集合）来存储请求记录。ZSet 中每个元素包含两个部分：**member 存储请求的唯一标识（如 UUID），score 存储请求到达的时间戳（毫秒级）**。由于 ZSet 会按照 score 自动排序，这天然适合按时间顺序管理请求记录。

**Lua 脚本的执行流程如下：**

首先，脚本会根据当前时间和窗口大小计算出窗口的起始时间，然后调用 `ZREMRANGEBYSCORE` 命令删除所有 score 小于起始时间的元素。这一步是**清理过期**数据，保证窗口内只保留最近一段时间内的请求记录。

接着，通过 **ZCARD** 命令统计当前 ZSet 中的元素个数，这个数字就是窗口内的总请求数。将统计结果与设定的阈值进行比较。

如果当前请求数小于阈值，说明未超过限流限制，脚本会调用 ZADD 命令将当前请求插入 ZSet，member 使用请求唯一标识，score 使用当前时间戳。同时设置 EXPIRE 过期时间，避免 ZSet 无限增长占用过多内存。最后返回成功标识，表示请求放行。

如果当前请求数已经达到或超过阈值，脚本直接返回失败标识，表示请求被限流拦截。

整个流程中，删除过期数据、统计数量和插入新请求这三个操作在一个 Lua 脚本中原子执行。Redis 的单线程特性保证了脚本执行期间不会被其他请求打断，从而避免了并发场景下的统计误差。

### 1.4 注解与 AOP 的集成

为了让限流组件无侵入且易于使用，采用自定义注解加 AOP 切面的方式实现。

首先定义一个 `@RateLimit` 注解，包含以下几个关键属性：限流 key 的前缀、时间窗口大小、窗口内允许的请求数阈值、限流被触发时的提示信息，以及限流维度（全局、用户或 IP）。有了这个注解后，只需要在需要限流的接口方法上添加注解并配置相应参数即可。

接着实现一个 AOP 切面，拦截所有带有 `@RateLimit` 注解的方法。切面中会根据注解参数和当前请求上下文动态构建 Redis key。如果是用户维度的限流，key 中会包含用户 ID；如果是 IP 维度的限流，key 中会包含客户端 IP；如果是全局限流，key 中则只包含**接口标识**。构建好 key 后，切面调用限流 Lua 脚本，将窗口大小、阈值、当前时间戳等参数传递给脚本执行。根据脚本返回的结果，决定是放行请求继续执行原方法，还是抛出限流异常返回提示信息。

### 1.5 维度隔离的实现

三种限流维度通过不同的 key 命名规则实现隔离：

全局限流使用 `limit:global:{接口标识}` 作为 key，所有请求共享同一个计数器。用户限流使用 `limit:user:{用户ID}:{接口标识}`，每个用户拥有独立的计数器，互不影响。IP限流使用 `limit:ip:{客户端IP}:{接口标识}`，每个 IP 独立计数。

这种设计使得不同维度的限流规则可以同时生效且互不干扰。例如，秒杀接口可以同时配置全局限流防止系统过载，又配置用户限流防止单个用户刷单，两层保护叠加，既保证了系统稳定性，又维护了公平性。

## 二、限流算法
### 2.1 固定窗口限流(又称计数器算法)
首先维护一个计数器，将单位时间段当做一个窗口，计数器记录这个窗口接收请求的次数。

+ 当次数少于限流阀值，就允许访问，并且计数器+1
+ 当次数大于限流阀值，就拒绝访问。
+ 当前的时间窗口过去之后，计数器清零。

假设单位时间是1秒，限流阀值为3。在单位时间1秒内，每来一个请求,计数器就加1，如果计数器累加的次数超过限流阀值3，后续的请求全部拒绝。等到1s结束后，计数器清0，重新开始计数。

![](https://jiwang.cc.cd/PicGo/project/youxiang/2026-1774282634-1d5645.png)

这种算法有一个很明显的<font style="color:rgb(216,57,49);">临界问题</font>：假设限流阀值为5个请求，单位时间窗口是1s,如果我们在单位时间内的前0.8-1s和1-1.2s，分别并发5个请求。虽然都没有超过阀值，但是如果算0.8-1.2s,则并发数高达10，已经超过单位时间1s不超过5阀值的定义啦。

### 2.2 滑动窗口限流
滑动窗口限流解决固定窗口临界值的问题。它将单位时间周期分为n个小周期，分别记录每个小周期内接口的访问次数，并且根据时间滑动删除过期的小周期。

<!-- 这是一张图片，ocr 内容为： -->
![](https://jiwang.cc.cd/PicGo/project/youxiang/2026-1774282634-d3253c.png)

滑动窗口的格子周期划分的越多，那么滑动窗口的滚动就越平滑，限流的统计就会越精确。

滑动窗口算法虽然解决了固定窗口的临界问题，但是一旦到达限流后，请求都会直接暴力被拒绝。酱紫我们会损失一部分请求，这其实对于产品来说，并不太友好。

### 2.3 漏桶算法
漏桶算法面对限流，就更加的柔性，不存在直接的粗暴拒绝。

它的原理很简单，可以认为就是注水漏水的过程。往漏桶中以任意速率流入水，以固定的速率流出水。当水超过桶的容量时，会被溢出，也就是被丢弃。因为桶容量是不变的，保证了整体的速率。

<!-- 这是一张图片，ocr 内容为： -->
![](https://jiwang.cc.cd/PicGo/project/youxiang/2026-1774282634-288c16.png)

+ 流入的水滴，可以看作是访问系统的请求，这个流入速率是不确定的。
+ 桶的容量一般表示系统所能处理的请求数。
+ 如果桶的容量满了，就达到限流的阀值，就会丢弃水滴（拒绝请求）
+ 流出的水滴，是恒定过滤的，对应服务按照固定的速率处理请求。

在正常流量的时候，系统按照固定的速率处理请求，是我们想要的。但是<font style="color:rgb(216,57,49);">面对突发流量的时候，漏桶算法还是循规蹈矩地处理请求，这就不是我们想看到的啦。</font>流量变突发时，我们希望系统尽快处理请求，提升用户体验。

### 2.4 令牌桶算法
面对突发流量的时候，我们可以使用令牌桶算法限流。

+ 有一个令牌管理员，根据限流大小，定速往令牌桶里放令牌。
+ 如果令牌数量满了，超过令牌桶容量的限制，那就丢弃。
+ 系统在接受到一个用户请求时，都会先去令牌桶要一个令牌。如果拿到令牌，那么就处理这个请求的业务逻辑；
+ 如果拿不到令牌，就直接拒绝这个请求。

如果令牌发放的策略正确，这个系统即不会被拖垮，也能提高机器的利用率。

## 三、代码实现
### 3.1 核心思路

限流组件的本质是：**在方法执行前，根据配置的规则检查是否超过限制，超过则拦截，否则放行。**

整个实现分为三层：

1. **注解层**：定义限流规则（窗口多大、限制多少次、按什么维度限制）
2. **切面层**：拦截带注解的方法，构建Redis Key，调用Lua脚本
3. **脚本层**：用ZSet实现滑动窗口计数，返回是否放行

------

### 3.2 自定义注解：定义限流规则

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface RateLimiter {
    // 时间窗口（秒）
    int window() default 10;
    
    // 窗口内允许的请求数
    int limit() default 20;
    
    // 限流维度：IP / 用户 / 全局
    LimitType type() default LimitType.GLOBAL;
    
    // 被限流时的提示信息
    String message() default "请求过于频繁，请稍后再试";
    
    enum LimitType {
        IP,    // 按IP限流
        USER,  // 按用户限流
        GLOBAL // 全局限流
    }
}
```

```java
@RateLimiter(window = 1, limit = 5, type = LimitType.IP)
public String getMerchantInfo() { ... }
```

### 3.3 切面类：核心拦截逻辑

```java
@Aspect
@Component
public class RateLimiterAspect {
    
    @Resource
    private StringRedisTemplate redisTemplate;
    
    // 加载Lua脚本
    private static final DefaultRedisScript<Long> SCRIPT = new DefaultRedisScript<>();
    static {
        SCRIPT.setLocation(new ClassPathResource("limiter.lua"));
        SCRIPT.setResultType(Long.class);
    }
    
    @Before("@annotation(rateLimiter)")
    public void before(JoinPoint point, RateLimiter rateLimiter) {
        // 1. 获取注解参数
        int window = rateLimiter.window();
        int limit = rateLimiter.limit();
        
        // 2. 构建Redis Key（区分不同维度）
        String key = buildKey(point, rateLimiter);
        
        // 3. 执行Lua脚本，返回0表示被限流
        Long result = redisTemplate.execute(
            SCRIPT,
            Collections.singletonList(key),
            String.valueOf(window),
            String.valueOf(limit),
            String.valueOf(System.currentTimeMillis())
        );
        
        // 4. 被限流则抛异常
        if (result != null && result == 0) {
            throw new RateLimitException(rateLimiter.message());
        }
    }
    
    // 构建Key：根据维度拼接不同前缀
    private String buildKey(JoinPoint point, RateLimiter rateLimiter) {
        MethodSignature sig = (MethodSignature) point.getSignature();
        String methodPath = sig.getMethod().getDeclaringClass().getName() 
                          + ":" + sig.getMethod().getName();
        
        switch (rateLimiter.type()) {
            case IP:
                return "limit:ip:" + getClientIp() + ":" + methodPath;
            case USER:
                return "limit:user:" + getCurrentUserId() + ":" + methodPath;
            default:
                return "limit:global:" + methodPath;
        }
    }
    
    private String getClientIp() {
        // 从Request中获取真实IP（代码略）
    }
    
    private String getCurrentUserId() {
        // 从SecurityContext或Session中获取当前用户ID（代码略）
    }
}
```

### 3.4 Lua脚本：滑动窗口计数

```lua
-- KEYS[1]: 限流key
-- ARGV[1]: 窗口大小（秒）
-- ARGV[2]: 限制次数
-- ARGV[3]: 当前时间戳（毫秒）

local key = KEYS[1]
local window = tonumber(ARGV[1]) * 1000  -- 秒转毫秒
local limit = tonumber(ARGV[2])
local now = tonumber(ARGV[3])

-- 1. 删除窗口外的请求（score < now - window）
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

-- 2. 统计窗口内请求数
local current = redis.call('ZCARD', key)

-- 3. 判断是否放行
if current < limit then
    -- 放行：记录本次请求
    redis.call('ZADD', key, now, now .. '-' .. math.random(1000000))
    redis.call('EXPIRE', key, window / 1000)
    return 1
else
    -- 限流
    return 0
end
```

### 3.5 异常处理

```java
// 自定义异常
public class RateLimitException extends RuntimeException {
    public RateLimitException(String message) {
        super(message);
    }
}

// 全局捕获，返回友好提示
@RestControllerAdvice
public class GlobalExceptionHandler {
    @ExceptionHandler(RateLimitException.class)
    public Result handleRateLimit(RateLimitException e) {
        return Result.fail(e.getMessage());
    }
}
```
