## 二、rag实现
这部分主要解决的是**向量数据库与业务数据库之间的数据一致性问题**。

类似的项目经验不多，因此这套方案也是在**探索中逐步形成**的。核心目标是确保像商户表这类关键数据，能及时同步到向量数据库，从而让用户获取到准确的信息，避免出现店铺已新增却无法通过 RAG 获取、或店铺已删除却仍被检索到的情况。

要保证一致性，实际上需要解决的就是对增、删、改操作的检测与数据同步。实现上主要依托定时任务，具体细节可参考 AI 实现部分。这种方式相较静态文档读取更灵活，也支持直接读取数据库，毕竟需要保证实时性的表并不多。

这段逻辑的实现方式是：首先查出最近一轮发生新增或修改的向量，直接通过检查更新时间字段即可筛选出来。然后针对每条数据，在向量库中查询是否存在与对应商户 ID 相关联的向量，具体通过 `inMemoryStore.remove(shopIdStr)` 方法处理——如果该 ID 不存在，方法不会报错，只是不做任何操作；如果存在，则直接将其删除。

为了解决业务数据库删除导致的向量数据库和业务数据库数据不一致的问题，在向量管理中，我设计了一个 <font style="background-color:#81BBF8;">映射表</font> 来记录商铺ID和向量ID的对应关系。<font style="background-color:rgba(183,237,177,0.8);">每次增量更新时</font>，先处理删除的情况，对比映射表和数据库，找出已被删除的商铺，然后清理对应的向量和映射表记录。

![](https://jiwang.cc.cd/PicGo/project/youxiang/2026-1774282311-0090d9.png)

**映射表**就是一个“**小本本**”，记着每个商铺 ID 对应哪个向量 ID，这样你要删商铺的时候，翻一下小本本就能找到对应的向量把它删掉。

**映射表的更新过程**，就是往向量库里存数据的时候，顺手在小本本上记一笔“这个商铺 ID 对应的是那个向量 ID”；删数据的时候，先翻小本本找到向量 ID，删掉向量，再擦掉小本本上这笔记录。增、删、改都同步维护，保证小本本和真实数据始终一致。

如果是“改”的情况，更新商铺信息时，先按原来的向量 ID **删**掉旧向量，再生成新向量**存**入向量库，然后在小本本上把对应的向量 ID 更新成新的。

这样就实现了对新增、更新、删除三种情况的完整支持。
