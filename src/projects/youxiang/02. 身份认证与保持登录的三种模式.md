## 一、Cookie + Session（传统模式）
<img src="https://jiwang.cc.cd/PicGo/project/youxiang/2026-1774282575-a67254.png" alt="image.png" style="zoom:80%;" />

具体流程：

> 1. 用户输入用户名密码提交登录
> 2. 服务器验证通过，创建Session并存储用户信息
> 3. 服务器响应：`Set-Cookie: JSESSIONID=abc123; HttpOnly`
> 4. 浏览器存储Cookie
> 5. 后续请求自动携带：`Cookie: JSESSIONID=abc123`
> 6. 服务器通过SessionID查找Session，验证用户身份
>

<font style="color:rgb(216,57,49);">其中JSESSIONID 是 Java Web 容器默认用于保存 SessionID 的 Cookie 名称，服务器通过它找到对应的 Session 对象，从而识别用户身份。</font>

**Session 的保持时间**主要由服务端的 Session 超时时间和客户端的 Cookie 有效期共同决定，两者取**较小值**作为实际有效时间。

服务端 Session 超时时间默认为 30 分钟，表示用户如果在 30 分钟内没有发起任何请求，Session 就会被服务器销毁。

客户端携带 JSESSIONID 的 Cookie 默认是会话级 Cookie，浏览器关闭即失效。

若需要更长的登录保持（如“记住我”功能），可以同时延长 Session 超时时间并设置 Cookie 的 Max-Age 有效期（如 7 天），这样在有效期内即使用户关闭浏览器再打开，依然能保持登录状态。

## 二、Token方案
<img src="https://jiwang.cc.cd/PicGo/project/youxiang/2026-1774327182-9ae253.png" style="zoom:80%;" />

Token流程细节（以JWT为例）：

> 1. 客户端提交认证信息
> 2. 服务器验证成功后生成JWT：
>
>     Header.Payload.Signature 
>
>     返回JWT给客户端
>    
> 3. 客户端存储JWT
>    4. 客户端在后续请求中通过 HTTP Header 自动携带 JWT。
> 5. 服务器验证 JWT 签名，若通过，则解析 Payload 获取用户身份信息，并进行具体的 HTTP 请求处理。
> 

### 2.1 JWT三部分详解

#### （1）Header（头部）

**作用**：描述JWT的元数据，通常包含令牌类型和签名算法。
**内容示例**：

```json
{
  "alg": "HS256",   // 签名算法，HMAC SHA256
  "typ": "JWT"      // 令牌类型
}
```

**处理**：将Header进行**Base64Url编码**，得到第一部分。

#### （2）Payload（负载）

**作用**：存放实际需要传递的声明（Claims），包括用户信息、过期时间等。
**内容示例**：

```json
{
  "sub": "1234567890",        // 用户唯一标识
  "name": "John Doe",         // 自定义用户信息
  "iat": 1516239022,          // 签发时间（issued at）
  "exp": 1516242622,          // 过期时间（expiration）
  "role": "admin"             // 自定义字段
}
```

**注意**：Payload中**不要存放敏感信息**（如密码），因为JWT的Payload仅是Base64编码，并非加密，可被轻易解码查看。
**处理**：将Payload进行**Base64Url编码**，得到第二部分。

------

#### （3）Signature（签名）

**作用**：确保JWT未被篡改，由服务器使用密钥对前两部分进行签名生成。
**生成方式**（以HS256为例）：

```text
signature = HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret_key
)
```

- **HMACSHA256**：一种基于哈希的消息认证码算法，需要密钥。
- **secret_key**：仅服务器持有的密钥（例如 `"my_secret_key_123"`）。

将生成的签名再进行**Base64Url编码**，得到第三部分。

最终JWT格式：

```
		xxxxx.yyyyy.zzzzz
（Header编码.Payload编码.Signature编码）
```

### 2.2 JWT首次生成流程（服务器端）

1. **接收认证信息**：用户提交用户名/密码，服务器验证通过。
2. **构建Header**：定义使用的算法（如HS256）和类型（JWT）。
3. **构建Payload**：放入用户标识（sub）、过期时间（exp）等必要声明。
4. **计算签名**：使用`HMACSHA256(headerEncoded + "." + payloadEncoded, secretKey)`生成签名。
5. **拼接返回**：将三部分用`.`连接，返回给客户端。

------

### 2.3 JWT验证流程（服务器端）

当客户端在后续请求的`Authorization: Bearer <JWT>`中携带令牌时：

1. **分离三部分**：按`.`分割，得到headerEncoded、payloadEncoded、signatureReceived。
2. **重新计算签名**：使用**相同的密钥**和`headerEncoded + "." + payloadEncoded`，再次通过HMACSHA256计算出一个新签名。
3. **比对签名**：
   - 若新签名 ≠ signatureReceived → **验证失败**，返回401。
   - 若相等 → 签名有效，说明令牌未被篡改。
4. **解析Payload**：Base64Url解码Payload，提取用户ID、过期时间等信息。
5. **校验过期时间**：对比`exp`声明与当前服务器时间，若已过期则拒绝。
6. **处理请求**：通过用户身份信息（如`sub`）继续执行业务逻辑。

### 2.4 非对称加密的数学关系

#### （1）密钥对的生成原理

非对称加密（以RSA为例）的核心在于利用大数分解的数学难题。整个密钥对并非独立生成，而是通过一套数学算法同时产生。

首先生成两个非常大的质数p和q，这两个质数通常有数百位十进制数字，且需要严格保密。随后计算它们的乘积n = p × q，n作为模数会成为公钥和私钥的共同组成部分。接着计算欧拉函数φ(n) = (p-1) × (q-1)，并选择一个与φ(n)互质的整数e作为**公钥指数**。最后通过扩展欧几里得算法计算出**私钥指数**d，使得(e × d) mod φ(n) = 1。至此，密钥对生成完毕：**公钥为(n, e)，私钥为(n, d)**。整个过程确保公钥和私钥在数学上形成一对一的配对关系，任何一方的运算结果都可以被另一方还原。

#### （2）公私钥的推导关系

从数学角度看，私钥可以推导出公钥。因为私钥中包含n和d，而持有私钥意味着知道p和q（或至少知道φ(n)），因此可以计算出与d配对的e，从而得到完整的公钥。反之，从公钥(n, e)推导私钥(n, d)在计算上极其困难。要得到d，必须知道φ(n)，而φ(n)需要分解n得到p和q。当n足够大（如2048位）时，分解n需要超级计算机耗费数十年乃至更长时间，这在当前的计算能力下被认为是不可能完成的任务。这种单向性正是非对称加密安全性的根本保障。

#### （3）数学上的配对特性

公钥和私钥在数学上存在严格的互逆关系。对于任意消息m，用公钥加密后得到密文c = m^e mod n，再用私钥解密即可恢复原消息m = c^d mod n。同样的道理，如果先用私钥进行签名运算得到s = m^d mod n，那么用公钥验证时，计算s^e mod n即可还原m。这种互逆性保证了一个密钥对可以用于加密/解密和签名/验证两种场景。**在数字签名中，私钥用于生成签名，公钥用于验证签名；在数据加密中，公钥用于加密，私钥用于解密。**两者的角色虽然不同，但底层依赖的是同一套数学关系。

#### （4）与对称加密的本质区别

对称加密使用同一个密钥进行加密和解密，其安全性依赖于密钥的保密性，密钥必须通过安全信道预先分发给多个服务。而非对称加密通过公私钥分离解决了密钥分发问题：公钥可以公开传播，私钥由持有者妥善保管。对称加密的数学基础通常是混淆和扩散操作（如AES中的S盒和P盒），而非对称加密依赖数论难题（如大数分解、离散对数）。在性能上，对称加密比非对称加密快数百倍到数千倍，因此实际应用中常采用混合加密方案：用非对称加密传递对称密钥，再用对称密钥加密实际数据。

## 三、Session + Token混合
<img src="https://jiwang.cc.cd/PicGo/project/youxiang/2026-1774327104-8c8f6f.png" style="zoom: 67%;" />

在使用 **JWT + Refresh Token** 的登录体系中，Refresh Token 的可选刷新机制有两个核心作用：<font style="color:rgb(216,57,49);">长期无缝登录；提升安全性</font>。

<font color="red">可选刷新机制的内涵</font>：

当客户端用 Refresh Token 换 Access Token 时，服务器会检查 Refresh Token 剩余有效期，如果剩余有效期很短，比如只剩三天，服务器同时生成一个新的 Refresh Token 返回给客户端，旧的 Refresh Token 作废；如果剩余有效期还够长，继续使用原来的 Refresh Token。

注意，这里服务器生成新 Refresh Token 不是无缘无故自己刷新，而是响应客户端用 Refresh Token 换 Access Token 的请求时顺便做的，也就是说，刷新是<font color="blue">被动触发</font>的。

## 四、本章小结

本文梳理了三种主流的用户认证方案，从传统的`Cookie+Session`到无状态的`JWT Token`，再到兼顾安全与体验的混合模式，呈现了认证技术演进的核心脉络。

Cookie+Session作为经典的认证方式，其核心在于服务端维护会话状态，客户端仅持有一个会话标识。这种方案的优点在于实现简单、注销可控，服务端可以随时销毁Session实现立即失效。但其缺点同样明显：在分布式架构下需要解决Session共享问题，且会话状态占用服务端内存，影响横向扩展能力。尽管如此，对于**单体应用或用户规模不大**的系统，这仍然是一种可靠的选择。

JWT Token方案则代表了无状态认证的典型实践。服务端生成包含用户信息和签名的自包含令牌，客户端在后续请求中携带该令牌，服务端通过验证签名即可确认用户身份，无需存储任何会话信息。这种设计天然支持水平扩展，尤其适合微服务架构。以HS256为代表的对称签名方式使用单一密钥完成签名和验证，实现简单；而以RS256为代表的非对称签名则通过公私钥分离，解决了密钥分发的安全性问题，让认证服务与业务服务解耦成为可能。非对称加密的数学基础在于**大数分解难题**，私钥可以推导出公钥，但反之则不可行，这种**单向性**保障了整个体系的安全。

然而，纯JWT方案也存在明显短板：Token一旦签发，在有效期内无法主动失效，且Payload中不宜存放敏感信息。为解决这些问题，混合方案引入了Refresh Token机制，将认证凭证拆分为短期有效的Access Token和长期有效的Refresh Token。Access Token用于日常业务请求，即使泄露影响范围也有限；Refresh Token则仅在换取新Token时使用，且采用被动刷新的策略，在换取新Access Token时同步评估Refresh Token的剩余有效期，仅在接近过期时才颁发新的Refresh Token，避免了服务端主动刷新的开销。这种设计既实现了长期无缝登录的用户体验，又通过短期令牌机制提升了安全性。

:::warning 适用场景
Cookie+Session适合传统单体应用，纯JWT适合分布式无状态架构，混合模式则在需要兼顾安全性与用户体验的场景下展现出优势。在实际选型时，应根据系统的架构形态、安全要求、用户规模等因素综合权衡，选择最适合的方案。
:::
